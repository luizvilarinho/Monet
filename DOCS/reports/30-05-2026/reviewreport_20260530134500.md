## src/components/SearchPalette/SearchPalette.tsx

### Correção
- Discriminated union `SearchResult` está bem tipada. A função `selectResult` centraliza o dispatch para notebook ou nota — sem duplicação.
- `sortResults` ordena por: notebook-name match (0) → title match (1) → outros (2). Correto conforme spec.
- Resultados de notebook aparecem acima dos de nota — correto.
- `subjectId` é passado corretamente via `note.subjectId`.

### Stale closure em `useEffect` de busca

**[importante]** O `useEffect` de busca (linha 56) declara `[query]` como dep, mas usa `notebooks` e `notebookMap` dentro do closure. Se `notebooks` mudar enquanto o painel está aberto, os resultados de notebook podem estar desatualizados até que o usuário mude o `query`. Na prática, notebooks raramente mudam enquanto o palette está aberto, mas o dep array está tecnicamente incompleto. A correção seria adicionar `notebooks` (ou a referência estabilizada de `notebookMap`) ao dep array.

### Sem o `selectResult` no dep array do `useEffect` do teclado

**[nit]** O `useEffect` de keydown (linha 89) lista `onSelectNote` e `onSelectNotebook` no dep array mas não lista `selectResult`. Como `selectResult` é uma função definida no corpo do componente (re-criada a cada render), isso cria uma stale closure de menor impacto — o `selectResult` no closure do keydown pode usar referências antigas de `results`. Pelo fato de `results` já estar no dep array, o efeito reinicia quando os resultados mudam, então a janela de staleness é pequena. Não é bloqueante.

## src/components/CalendarView/CalendarView.tsx

### Correção
- `dayDotMap` consolida os dois `Set` anteriores em um único `Map<string, 'todo' | 'content'>`. A prioridade `'todo'` sobre `'content'` está correta — uma nota com TODO pendente não deve perder o dot roxo por causa de outra nota de tipo `'content'` no mesmo dia.
- A verificação `noteHasContent(content.trim().length > 0)` captura corretamente notas com apenas whitespace como "vazias". **Atenção**: o conteúdo é armazenado como Markdown — um Tiptap "empty document" serializado pode gerar `""` ou apenas `"\n"`, ambos capturados pela verificação. Se o Tiptap serializar um parágrafo vazio como `\n`, o dot não aparecerá — comportamento correto conforme spec.
- `noteHasUnfinishedTodos` usa `/^- \[ \] .+/m`. O padrão é correto para Markdown gerado pelo Tiptap (task list serializada como `- [ ] texto`). Funciona.

### Formato de chave — potencial miss

**[importante]** No `dayDotMap`, a primeira loop usa `note.title` (formato `dd/mm/yyyy`) como chave. Na renderização, o lookup faz `dayDotMap.get(title) ?? dayDotMap.get(dateKey)` onde `title = dayTitle(cell.day)` (também `dd/mm/yyyy`) e `dateKey = toDateKey(year, month, cell.day)` (formato `yyyy-mm-dd`). O segundo loop (`datedNotes`) usa `note.date` como chave — mas `note.date` pode estar no formato `yyyy-mm-dd` (típico de campos de data ISO). Se `note.date` for `yyyy-mm-dd`, o lookup pelo `dateKey` (`yyyy-mm-dd`) vai bater corretamente. Porém, se `note.date` estiver em outro formato (ex.: `dd/mm/yyyy`), o lookup pelo `dateKey` falha silenciosamente. Isso é comportamento existente preservado (a lógica anterior também dependia de `note.date`). Sem contexto suficiente para afirmar o formato exato de `note.date` — registrado como observação.

## src/components/Editor/CommandExtension.ts

### `computeTaskDropTarget`

A lógica de walk up via `$drop.depth` para encontrar o `taskItem` ancestral está correta. O clamp final `Math.max(listPos + 1, Math.min(dropResult.pos, listEnd - 1))` é um fallback razoável. A detecção de "antes/depois" pelo meio vertical do rect é a abordagem padrão.

### `makeTaskDragHandle` — uso de `taskItemPos` vs `initialFrom`

**[importante]** O handle captura `taskItemPos` (posição no momento da criação da decoração) em `initialFrom`. Em `onMouseUp`, ele chama `view.state.doc.nodeAt(initialFrom)` para obter o nó atual. O problema: se o documento foi editado entre a criação da decoração e o início do drag (ex.: o usuário digitou em outro item antes de arrastar), `initialFrom` pode estar desatualizado — o nó no `initialFrom` pode não ser mais o `taskItem` original, ou pode ser um nó diferente. O código verifica `currentNode.type.name === 'taskItem'`, o que previne corrupção óbvia, mas pode silenciosamente não executar o drop se o item se deslocou. Em uso normal (usuário arrasta logo após a decoração ser criada), isso não é problema. É uma limitação inerente do padrão usado para o drag handle de blocos de comando existente — não é um bug novo introduzido aqui.

### Decorações `taskItem` dentro do `if (view)`

A segunda passagem `state.doc.descendants` para `taskItem` está dentro do bloco `if (view)` — correto, o drag handle requer a view. Retorna `false` para não descer dentro do `taskItem` (os filhos do `taskItem` não precisam de handles). Correto.

### `key: taskdrag-${pos}`

A chave usa a posição absoluta no documento. Se o documento muda (outros itens adicionados antes), as posições mudam e as decorações são recriadas na próxima transação — comportamento correto porque `buildDecorations` é chamado em cada `apply`.
