# Task: Notes Area — 9 Adjustments

## Objective
Apply 7 incremental improvements and fix 2 bugs across the Notes area of the Monet app. Items cover drag-and-drop ordering, advanced search UX, Calendar note visibility, note breadcrumb display, spell-check context menu, calendar dot indicators, and two slash command changes.

## Context
- Module: Notes (all notes list, notebooks, subjects, calendar notebook, Tiptap editor, advanced search/ctrl-p)
- Stack: Tauri + React + Tiptap; drag-and-drop via manual mousedown/mousemove/mouseup (HTML5 DnD does not work inside contenteditable)
- Calendar notebook is a special notebook persisted by ID in localStorage

## Requirements

### 1 — Drag handle for TODO items
- Blocks inserted via slash command already show a drag handle icon on the left side for reordering
- Extend the same drag handle to TODO (task list) items so they can be reordered the same way

### 2 — Advanced search (Ctrl+P): prioritized match display
- Sort results: notebook name match → note name match → body/other matches
- For each result, show the notebook name discretely (e.g., small muted label above the note title) so notes with the same name are distinguishable

### 3 — All Notes: hide Calendar notes
- Calendar notes pollute the "All Notes" list; group them under a collapsible folder/section labeled "Calendar" with a toggle to show/hide
- If technically infeasible, remove Calendar notes entirely from the list

### 4 — BUG: Subject name missing in note header (All Notes context)
- When a note is opened from "All Notes" and it belongs to a Subject inside a notebook, the breadcrumb at the top should show: Notebook › Subject › Note
- Currently the Subject segment is omitted; behavior should match opening the note from the notebook's own list

### 5 — BUG: Spell-check context menu broken
- Right-clicking a spell-checked (underlined) word no longer shows correction suggestions — only the "Link notes" option appears
- Should restore correction suggestions in the context menu alongside other options

### 6 — Calendar: red dot only when note has content
- Currently a red dot appears on a calendar day as soon as a note is created, even if it is empty
- Dot should only appear when the note has meaningful content (non-empty body)

### 7 — Calendar: purple dot for notes with incomplete TODOs
- If a calendar day's note contains at least one unchecked TODO item, show a purple dot (in addition to or replacing the red dot)
- Red dot = has content; purple dot = has unchecked TODOs (priority indicator)

### 8 — Slash command: /action
- Add a new `/action` slash command that instructs the AI model to perform an action on the selected/surrounding text (edit, rewrite, organize, transform, etc.)
- Should follow the pattern of existing slash commands in the editor

### 9 — Slash command: verify /mindmap naming
- Check whether the `/mindmap` slash command should be renamed to `/roadmap` based on its actual behavior/output
- If it generates a roadmap-style output, rename it; otherwise keep the name

## Constraints
- Item 1: extend the existing drag handle mechanism (already works for slash command blocks) to TODO nodes — do not reimplement from scratch
- No breaking changes to existing note data structures unless scope explicitly allows it
- Calendar notebook ID is persisted in localStorage — changes must not invalidate existing data
- Do not add unnecessary abstractions; match the code style already in place

## Edge Cases
- Item 2: a note with no notebook context (edge case in search results)
- Item 3: user may have Calendar as their active/selected notebook — toggling hide should not break navigation
- Item 4: notes that belong directly to a notebook (no subject) must still show correctly
- Item 6: "has content" must handle Tiptap's default empty paragraph node (not just empty string)
- Item 7: purple dot should not appear if all TODOs in the note are checked
- Item 9: if mindmap and roadmap are different concepts in the codebase, both may need to coexist
