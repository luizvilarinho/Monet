## Análise por arquivo

### `src-tauri/src/lib.rs`

**Correção**

- `extract_pdf_text` usa `spawn_blocking` corretamente para não bloquear o runtime Tokio. A captura de pânico via `e.is_panic()` é adequada para uma crate de terceiros potencialmente não-panic-safe.
- `save_chat_doc`, `read_chat_doc` e `delete_chat_doc` são marcados como `async` mas usam `std::fs` síncrono. Para arquivos pequenos (texto extraído de PDFs/TXTs) o impacto é baixo, mas tecnicamente bloqueia o thread async do Tauri. `[nit]`
- `read_chat_doc` aceita um `path: String` arbitrário do frontend e faz `read_to_string` sem verificar se o caminho está dentro de `chat-docs/`. Isso representa path traversal: qualquer código no frontend pode ler qualquer arquivo acessível ao processo. `[importante]`
- `delete_chat_doc` tem o mesmo problema: aceita path arbitrário e remove o arquivo sem validação de prefixo. `[importante]`
- Campos `StreamToolCallEvent`, `supports_tools` em `ModelInfo`, `supported_parameters` em `OpenRouterModel` e a lógica de tool use no streaming são novidades fora do escopo descrito no taskreport, mas estão bem encapsulados e não quebram nada existente.

**Simplicidade**

- `save_chat_doc` poderia retornar apenas o `filename` em vez do path absoluto e o frontend reconstruiria o path, o que eliminaria o problema de path traversal por design. Porém, como o path absoluto varia por sistema, retornar o path completo é pragmático — a validação de prefixo seria suficiente. `[nit]`

**Efeitos colaterais**

- A adição de `supports_tools` em `ModelInfo` e `supported_parameters` em `OpenRouterModel` altera o payload retornado por `openrouter_list_models`. Se o campo não existir na API do OpenRouter, `supported_parameters` fica `None` e `supports_tools` fica `false` — comportamento seguro.
- `StreamError::ToolCall` adicionado sem branch de cleanup de UI: o handler no frontend (`onOpenRouterToolCall`) é quem continua o fluxo. Se o evento `tool_call` não chegar (bug na rede), o streaming termina silenciosamente sem emitir `done` nem `error`, deixando `isStreaming = true` para sempre no frontend. `[importante]`

**Padrões**

- Estilo Rust e uso de `#[tauri::command]` seguem o padrão do arquivo.

**Segurança**

- Path traversal em `read_chat_doc` e `delete_chat_doc` — ver Correção acima. `[importante]`

### `src/hooks/useChat.ts`

**Correção**

- Persistência de documentos via `save_chat_doc` acontece **antes** do guard `hasOpenRouterKey()`. Se a chave não estiver configurada, o arquivo é salvo no disco mas a conversa falha — o arquivo nunca é limpo porque o `ChatMessage` não chega ao estado (o `failOnAssistant` é chamado e a mensagem do usuário já está no estado com `attachedDocs`, mas sem limpeza dos paths salvos). `[importante]`

  Fluxo atual: `persistedDocs` → `updateMessages(userMsg com attachedDocs)` → `hasOpenRouterKey()` → falha → `failOnAssistant` (não limpa arquivos).

  Consequência: arquivos `.txt` acumulam em `chat-docs/` sem referência válida no histórico quando a chave está ausente (edge case de primeiro uso).

- `toolCancelledRef.current` é resetado para `false` no início de `send()`. Porém, `toolCallHandlerRef.current` é setado para `null` no `cancel()`, o que efetivamente aborta o handler antes de executar o segundo stream. O check `if (toolCancelledRef.current)` após as operações de busca é correto como segunda barreira.

- O segundo stream iniciado pelo tool handler (`requestId2`) não passa `tools`, então um segundo tool call do modelo não seria tratado — o stream terminaria com `StreamError::ToolCall` e a UI ficaria em `isStreaming = true`. Isso é aceitável para o escopo atual (single tool call por turno), mas é um limitador não documentado. `[nit]`

- `historyForApi` agora é `async` e faz I/O de disco para cada mensagem com doc antes de verificar a API key. Para conversas longas com muitos docs, esse I/O acontece mesmo quando a send vai falhar logo depois. `[nit]`

- `send` usa `currentDate` computado no início da função. Se o deep research / web search demora (vários segundos), `currentDate` ainda reflete o momento correto da chamada — sem problema.

- A dependência `models` foi adicionada ao array de deps do `useCallback`, o que é correto.

**Simplicidade**

- O bloco de tool call handler dentro de `send` (~80 linhas inline) é complexo de rastrear. Extrair para uma função fora do `useCallback` seria mais legível, mas não é um bug. `[nit]`

**Efeitos colaterais**

- `saveConversations` antes retirava `imageDataUrl` para não persistir base64. No diff, o strip de `attachedDocs` foi removido (paths são pequenos e devem persistir). O strip de `imageDataUrl` continua. Correto.
- `deleteConversation` e `deleteFolder` agora limpam os arquivos antes de remover o estado. Correto. A limpeza usa `void ... .catch()` — erros silenciosos são aceitáveis aqui.
- A renomeação da variável destructurada de `_` para `_img` em `saveConversations` é cosmética e correta.

**Padrões**

- Uso de `useCallback`, `useRef`, `useState` segue o padrão existente.
- Comentários misturam português e inglês (padrão misto já existia no arquivo).

**Segurança**

- O path retornado por `save_chat_doc` é armazenado no localStorage via `saveConversations`. Um path absoluto do sistema de arquivos no localStorage é inofensivo para um app desktop, sem exposição externa.

### `src/components/ChatPanel/ChatPanel.tsx`

**Correção**

- `handlePickFile` itera os arquivos com `for...of` e chama `return` ao encontrar um arquivo inválido (DOCX ou tipo não suportado). Isso aborta o loop inteiro na primeira rejeição — arquivos válidos que vêm **depois** do inválido na seleção são silenciosamente descartados. `[importante]`

  Exemplo: usuário seleciona `[doc.txt, illegal.docx, outro.md]` — `doc.txt` é aceito, `illegal.docx` dispara `return`, `outro.md` nunca é processado.

  Correção sugerida: substituir `return` por `continue` e acumular o erro, exibindo-o ao final.

- Para PDF, o `reader.onload` é `async` mas erros de `invoke` são capturados no `try/catch` interno. O `setIsPdfProcessing(false)` está no `finally` — correto.

- `resolveFileType` tem dead code para DOCX (`application/vnd.openxmlformats-officedocument.wordprocessingml.document`): o early-return no loop usa `isDocx` baseado no mesmo mime. Não é bug, mas a constante de mime DOCX existe em dois lugares. `[nit]`

- Quando o usuário troca de conversa (`activeId` muda), `draftDocs` é resetado mas os arquivos eventualmente gerados durante o processamento de PDF (se o usuário trocar enquanto o PDF é extraído) podem ter sido salvos sem referência. Baixa probabilidade, aceitável. `[nit]`

- O botão Deep Research perdeu o atributo `disabled={!hasApiKey}` que o botão de imagem original tinha. O toggle de Deep Research pode ser ativado mesmo sem chave. Porém, a chave é verificada no `send()` antes de qualquer busca — impacto é só visual. `[nit]`

**Efeitos colaterais**

- `useChat()` passou a receber `models` como argumento: `useChat(models)`. Isso altera a assinatura do hook. A chamada foi atualizada corretamente no componente.
- `ChatToolsMenu` foi movido de posição no toolbar (agora após o botão Reasoning). Mudança cosmética intencional pelo taskreport.
- O `webSearchActive` é exibido como indicador no composer. A flag `setWebSearchActive(false)` é chamada dentro do tool handler após a busca concluir — há uma janela onde o usuário cancela *depois* da busca mas *antes* de `setWebSearchActive(false)`, deixando o indicador visível por instantes. Sem impacto real.

**Padrões**

- CSS modules, uso de `styles.*`, aria attributes, botões com `type="button"` — tudo segue o padrão.
- SVGs importados como módulo (`deep-gray.svg`, `deep-green.svg`) é o mesmo padrão usado para outros ícones no projeto.
- Linha em branco extra após o bloco de imports (linha 16 do diff em `ChatPanel.tsx`) — `[nit]`.
