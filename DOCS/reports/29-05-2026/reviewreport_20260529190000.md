### `src-tauri/src/documents.rs`

**Correção**
- A adição de `original_path` à struct `DocumentInfo` e ao SELECT de `documents_list_global` está correta. Os índices de coluna foram ajustados de 0–11 para 0–12 e estão certos.
- O emit de status `"indexing"` antes do scan (7a-2) está correto e resolve o feedback visual ausente.
- A propagação de visibilidade para arquivos novos segue a lógica descrita no executionreport: busca notebooks que já têm qualquer filho da pasta visível e insere o novo arquivo neles.

**Simplicidade**
- `[nit]` A lógica de propagação de visibilidade (`prepare → query_map → for loop`) poderia ser feita em uma única instrução SQL (`INSERT OR IGNORE INTO notebook_document_visibility SELECT DISTINCT notebook_id, ?new_id FROM notebook_document_visibility WHERE document_id IN (SELECT id FROM documents WHERE parent_folder_id = ?folder_id AND type = 'file')`). Nada crítico — o código atual funciona.

**Efeitos colaterais**
- `[importante]` A propagação de visibilidade usa `SELECT id FROM documents WHERE parent_folder_id = ?1 AND type = 'file'` **dentro da mesma transação em que o INSERT do novo arquivo acabou de ocorrer**. Isso significa que o novo arquivo já estará presente nessa subquery, o que é inócuo (o `INSERT OR IGNORE` vai ignorar se já existir), mas a query inclui o próprio `new_id_clone` nos notebooks encontrados. Em si, não causa bug — o `INSERT OR IGNORE` com o mesmo par `(notebook_id, new_id)` simplesmente não insere duplicata. No entanto, se a pasta ainda não tiver nenhum arquivo previamente visível (é a primeira vez que está sendo escaneada), `notebook_ids` estará vazio e nenhuma visibilidade é propagada — comportamento esperado.
- Sem efeitos colaterais no código existente para arquivos individuais.

**Padrões**
- Segue o padrão de `with_doc_db`, `params!`, `map_err(|e| format!(...))` consistente com o restante do arquivo.

**Segurança**
- Sem exposição de dados sensíveis.

**Organização e boas práticas**

### `src/hooks/useDocuments.ts`

**Correção**
- A lógica adicionada no handler de status detecta quando uma pasta transita para `'available'` e chama `refresh()` em vez de atualizar apenas o item local. Isso garante que a lista de filhos seja refletida após o scan.
- Correto e necessário, pois os filhos são inseridos no backend durante o scan e não chegam individualmente ao frontend via eventos de status.

**Efeitos colaterais**
- `[nit]` O `refresh()` é chamado toda vez que qualquer folder transita para `'available'`, mesmo que não haja filhos novos ou removidos. Em cenários com muitas pastas sendo rescaneadas em paralelo, isso pode gerar múltiplas requisições simultâneas ao backend. Aceitável dado o volume típico esperado.
