# Task: Watched Folders na Knowledge Base

## Objective
Adicionar suporte a pastas locais monitoradas na Knowledge Base. O usuário poderá cadastrar diretórios do sistema operacional e o app fará embedding automático dos arquivos contidos neles — detectando arquivos novos, modificados e removidos. O objetivo é reduzir o trabalho manual de adicionar documentos individualmente.

## Context
- Módulo afetado: Knowledge Base (RAG), incluindo UI e backend Tauri/Rust
- A KB hoje aceita somente arquivos individuais (type implicitamente "file")
- O banco vetorial é `monet-vec.db` (SQLite + embedding via `text-embedding-3-small`)
- A UI da KB e do modal "Notebook Documents" precisam ser atualizadas
- O scan de pastas deve ocorrer no startup do app e via botão manual na tela

## Requirements
- Adicionar botão `+ Add folder` na janela Knowledge Base que abre um seletor de diretório nativo (via Tauri dialog)
- A pasta cadastrada aparece como entrada única na lista com contador de arquivos detectados
- Múltiplas pastas podem ser cadastradas simultaneamente
- No startup e via botão manual: escanear cada pasta cadastrada, fazer embedding apenas de arquivos novos ou modificados desde o último scan, deletar chunks de arquivos que não existem mais na pasta
- Adicionar campo `type` (`file | folder`) na tabela da Knowledge Base
- Chunks continuam sendo indexados por arquivo individualmente, com referência ao folder pai
- Remoção de uma pasta deleta em cascata todos os chunks dos arquivos filhos
- No modal "Notebook Documents": pasta aparece como entrada única com toggle; ao ativar, todos os arquivos daquela pasta entram no contexto do caderno

## Constraints
- Não usar file watchers do sistema operacional (inotify, FSEvents, etc.) — verificação sob demanda apenas (startup + botão manual)
- Respeitar a arquitetura existente: comandos Tauri para operações de filesystem/banco, React para UI
- O campo `type` não deve quebrar a lógica existente de arquivos individuais
- Arquivos suportados devem seguir os mesmos critérios já aceitos pela KB atual
- Não reindexar arquivos que não mudaram (verificar por `last_modified` ou hash)

## Edge Cases
- Pasta vazia: cadastrar normalmente, exibir contador 0, não gerar erro
- Scan é recursivo: entrar em todos os subdiretórios da pasta selecionada
- Arquivos com nomes repetidos dentro da mesma pasta (em subpastas diferentes): o app deve usar o path completo como identificador único, não apenas o nome do arquivo
- Arquivo já cadastrado individualmente que também está numa pasta monitorada: re-indexar como entrada separada sob a pasta; são tratados como documentos distintos
- Ao deletar uma pasta monitorada: remover todos os arquivos embeddados originados dela do banco (cascade delete por folder_id)
- Pasta removida do disco após cadastro: marcar como inacessível, não crashar o app
- Pasta com muitos arquivos (ex.: 1000+): não bloquear a UI durante o scan/embedding (operação assíncrona)
- Remoção de arquivo da pasta entre dois scans: deletar chunks correspondentes do banco
- Permissão de leitura negada para um arquivo específico: pular e logar, não abortar o scan inteiro
