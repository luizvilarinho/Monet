# Plano

## Escopo

Adicionar suporte a **Watched Folders** na Knowledge Base do Monet. O usuário poderá cadastrar diretórios locais; o app fará embedding automático dos arquivos contidos neles, detectando novos, modificados e removidos. O scan ocorre no startup e via botão manual. Nenhuma breaking change nas funcionalidades existentes de arquivos individuais.

### Etapa 5 — Lógica de scan: função `scan_watched_folder_async`

- **Status:** PENDENTE
- **Descrição:** Implementar em `documents.rs` a função assíncrona interna `scan_watched_folder_async(app: AppHandle, folder_id: String)`. Esta é a função central da feature. Fluxo:

  **5a. Ler pasta:**
  - Buscar `original_path` e `status` do folder no DB.
  - Se a pasta não existir no disco: atualizar `status='error'`, `error_message='Folder not found on disk'`, emitir evento e retornar.

  **5b. Coletar arquivos em disco (recursivo):**
  - Usar `std::fs::read_dir` recursivamente. Em cada entrada:
    - Pular se `is_dir()` (recursão).
    - Pular se `detect_mime()` retornar `None` (tipo não suportado).
    - Pular se ocorrer `PermissionDenied` (logar e continuar).
    - Pular symlinks (`metadata.file_type().is_symlink()`).
  - Coletar lista de `(absolute_path: PathBuf, last_modified_ms: i64)`.
  - Usar `fs::metadata(path).modified()` para obter `last_modified_ms`.

  **5c. Buscar arquivos filhos já indexados (do folder):**
  - `SELECT id, original_path, last_modified_ms FROM documents WHERE parent_folder_id = ?1 AND type = 'file'`

  **5d. Detectar mudanças e agir:**
  - Para cada arquivo em disco:
    - Se NÃO existe no DB (por `original_path`): inserir novo documento (`type='file'`, `parent_folder_id=folder_id`, `status='indexing'`, etc.) e chamar `run_indexing` em spawn.
    - Se existe no DB e `last_modified_ms` diverge: atualizar `last_modified_ms`, marcar `status='indexing'`, apagar chunks via `delete_chunks_for_document`, chamar `run_indexing` em spawn.
    - Se existe e não mudou: pular.
  - Para cada arquivo no DB que NÃO aparece no disco: chamar `documents_delete_single(app, doc_id)` (helper interno que apaga chunks + visibilidade + row, sem tocar o arquivo físico — arquivos de pasta não têm cópia no `documents_dir`).

  **5e. Atualizar folder:**
  - Atualizar `status='available'` e `updated_at` do folder (apenas se não houve erro catastrófico).
  - Emitir evento `documents://status` para o folder com `status='available'`.

  **Observação sobre arquivos de pasta:**
  - Diferente de arquivos individuais (copiados para `documents_dir`), os arquivos de pasta são indexados diretamente do caminho original — sem cópia. O campo `original_path` armazena o path absoluto do arquivo no disco.
  - Por isso, ao deletar um arquivo filho de pasta, NÃO tentar apagar o arquivo físico (diferente de `documents_delete` atual).

  Registrar o comando público `documents_scan_watched_folder(app, state, folder_id: String)` que chama `scan_watched_folder_async` em spawn e retorna imediatamente `Ok(())`.
- **Arquivos:** `src-tauri/src/documents.rs`, `src-tauri/src/lib.rs`
- **Depende de:** Etapa 4

## Notas de implementação

- **Arquivos de pasta não são copiados** para `documents_dir`. O `original_path` de um arquivo filho aponta diretamente para o disco do usuário. Cuidado ao reutilizar `documents_delete` (que tenta `fs::remove_file`) — usar helper interno separado para filhos de pasta.
- **`documents_upload_global` não muda.** O campo `type` já tem `DEFAULT 'file'` na migration, então o INSERT existente continua funcionando sem alterar a query.
- **Atualização de status de folder:** o evento `documents://status` usa `document_id`, que para folders é o ID da pasta. O frontend já escuta e atualiza o item pelo id, portanto funciona sem mudança no listener.
- **Pasta removida do disco:** detectada no início do scan (5a), marca `status='error'` e retorna. O usuário pode remover manualmente ou tentar recriar a pasta.
- **Folder toggle no DocumentsModal:** usar `ref` em `<input>` para definir `indeterminate` via JS (atributo não disponível como prop React).
