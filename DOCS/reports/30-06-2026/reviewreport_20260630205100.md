## src/App.tsx
1. **Correção** —
   - `SYSTEM_PROMPT`: regra de comportamento para `/week` adicionada corretamente.
   - `buildUserMessage`: parâmetro `weekContext?` adicionado ao final da assinatura e ao array `parts`, seguindo o mesmo padrão de `ragContext`/`searchContext`.
   - Helpers novos (`formatCalendarTitle`, `parseCalendarTitle`, `stripEmbedBlockTags`, `getWeekCalendarNotes`, `formatWeekContext`) implementam corretamente os requisitos: janela de 7 dias inclusive a âncora (loop `i=6..0`), notas sem correspondência ou com conteúdo vazio omitidas silenciosamente (`continue`), `stripCommandLines` aplicado para não vazar a própria linha `/week` para o contexto, `stripEmbedBlockTags` remove tags de embed-block (não tenta expandi-las — ver observação abaixo).
   - `handleCommand`: guarda de defesa em profundidade (`def.calendarOnly && activeNote?.notebookId !== CALENDAR_NOTEBOOK_ID`) corretamente posicionada logo após a resolução de `def`. Erro tratado quando o título da nota ativa não é uma data válida. `notes` corretamente adicionado às dependências do `useCallback` (necessário para `getWeekCalendarNotes(notes, anchor)` não capturar um `notes` stale).
   - `handleCalendarDayClick` refatorado para usar `formatCalendarTitle`, eliminando duplicação de lógica de formatação de data — comportamento idêntico confirmado pela leitura do diff.
   - Prop `isCalendarNote={activeNote.notebookId === calendarNotebookId}` adicionada ao JSX do `Editor`, mesma comparação já usada para `notebookNotes`/`relatedContent` nas linhas vizinhas.
   - `tsc --noEmit -p .` rodado de forma independente: sem erros.

   **[importante] Semana sem nenhum conteúdo não gera erro nem contexto mínimo, mas o `SYSTEM_PROMPT` promete um bloco que simplesmente não existe na mensagem.**
   Se `getWeekCalendarNotes` retornar array vazio (nenhum dos 7 dias tem nota com conteúdo — cenário plausível: primeira nota de calendário do usuário, criada vazia, só com `/week pergunta`), `formatWeekContext` retorna `''`. Em `buildUserMessage`, `if (weekContext) parts.push(...)` descarta o bloco por ser string vazia, e como `noteContent` também é passado como `''` para `/week` (linha 716, de propósito, para não duplicar o conteúdo da nota atual), a mensagem final para o modelo fica sem **nenhum** bloco de contexto — só `Command: /week - ...` e `Parameter: ...`. O `SYSTEM_PROMPT`, porém, instrui o modelo a responder "usando exclusivamente o bloco 'Calendar notes from the last 7 days'" — bloco que nessa situação nem chega a existir na mensagem. O modelo fica sem sinal de que deveria dizer "não há informação suficiente" (a instrução do prompt presume que o bloco existe, ainda que vazio de conteúdo útil) e o risco é responder com conhecimento geral/alucinado sobre "a semana do usuário", que é exatamente o tipo de resposta que esse comando deveria evitar.
   Compare com o tratamento já existente para `/docs` no mesmo `handleCommand` (linhas 686-694): quando `ragContext` não é montado, o código bloqueia com `addErrorCard(...) ; return true` antes de chamar o modelo. Para `/week` esse guard equivalente não existe — nem no código, nem foi especificado no Plan Report (a omissão já vem do plano, o codeAgent seguiu fielmente). O Task Report cobre apenas omissão por dia ("omitir silenciosamente") e nota vazia isolada, mas não trata explicitamente o caso de a janela inteira ficar vazia.
   Sugestão de ajuste (não aplicada nesta review, é só code review): ou bloquear com `addErrorCard` quando `weekEntries.length === 0` (mesmo padrão do `/docs`), ou fazer `formatWeekContext` retornar um bloco explícito tipo `Calendar notes from the last 7 days: (none found)` para o modelo ter sinal claro de ausência de dados.

2. **Simplicidade** — Os helpers são simples e diretos, sem abstração desnecessária. `searchContext` é passado para `buildUserMessage` no branch `isWeekOnly` mas é sempre `undefined` para `/week` (já que `/week` não tem `usesSearch: true` em `commands.ts`, e `searchContext` só é populado dentro de `if (def.usesSearch && query.trim())`). `[nit]` Passar um parâmetro que nunca terá valor é levemente confuso, mas inofensivo e segue a ordem de assinatura de `buildUserMessage` — não vale a pena reestruturar por causa disso.
3. **Efeitos colaterais** — Nenhum comportamento de outros comandos foi alterado. O fluxo de RAG/`getNotebookVisibleDocumentIds` já rodava para qualquer comando executado em uma nota do notebook Calendar antes desta task (não havia restrição alguma por notebook previamente); `/week` está sujeito ao mesmo fluxo de RAG que qualquer outro comando rodando numa nota com notebook associado — não é uma regressão introduzida por este diff.
   `[nit]` `getWeekCalendarNotes` indexa notas do Calendar por título num `Map` (`byTitle`); se existirem duas notas com o mesmo título `DD/MM/YYYY` no notebook Calendar (cenário não impedido pela UI atual, mas tecnicamente possível se o usuário renomear manualmente uma nota para um título de data já existente), o `Map` mantém a última ocorrência da iteração de `notes`, que pode divergir de qual nota o `handleCalendarDayClick`/navegação do calendário trataria como "a" nota daquele dia (`notes.find` pega a primeira). Caso de borda raro, fora do que o Task Report pediu para tratar.
4. **Padrões** — Helpers novos posicionados junto das demais funções utilitárias de topo de arquivo (`stripCommandLines`, `expandEmbedBlocks`, `formatRagContext`), seguindo a mesma estrutura (formatação com cabeçalho + `---` final) de `formatRagContext`. Nomenclatura consistente com o restante do arquivo.
5. **Segurança** — Nenhuma exposição de dados sensíveis; conteúdo das notas concatenado no prompt segue o mesmo modelo de confiança já aceito para o restante do app (notas do próprio usuário, mesma superfície de "prompt injection" que já existe para `/summarize`, `/ask`, etc. — não é uma superfície nova).
6. **Organização** — Sem problemas de organização de arquivo/pastas; a lógica de `/week` está nos mesmos pontos onde `/docs` e os demais comandos já vivem, sem estrutura paralela.
