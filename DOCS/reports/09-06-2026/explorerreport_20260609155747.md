## [1] Remover Related Content da Agenda

### Localização do Componente
- **Component:** \src/components/RelatedContent/RelatedContent.tsx\ (linhas 22-75)
- **Props atuais:** \ctiveNote\, \

otebooks\, \onSelect\
- **Uso:** Renderizado em \src/App.tsx\ (linha 1048, aproximado)

### Mecanismo Atual
\\\
1. RelatedContent recebe a nota ativa
2. Chama scoreRelatedNotes() que usa:
   - Jaccard similarity entre tags, título e conteúdo
   - Pesos: tags (45%), título (25%), conteúdo (30%)
   - TOP-5 notas com score >= 0.04
3. Renderiza lista clicável que navega para notas
\\\

### Problema Identificado
- **Nota de agenda tem \	itle\ no formato \DD/MM/YYYY\** (vide \CalendarView.tsx\, linhas 174-178)
- Quando indexada, a data é tratada como conteúdo normal
- Resulta em "nearby dates" aparecerem na lista (ex: outras notas de agenda do mesmo mês)
- **Causa:** \scoreNotes.ts\ não filtra notas de agenda; apenas usa Jaccard similarity

### Solução (Baixa Dificuldade)
**Opção A (Recomendada):** Adicionar um parâmetro \calendarNotebookId\ ao \RelatedContent\ e excluir notas desse notebook da lista de candidatos.

\\\	ypescript
// Em scoreRelatedNotes(), linha 69:
if (note.id === active.id) continue
// Adicionar:
if (calendarNotebookId && note.notebookId === calendarNotebookId) continue
\\\

**Arquivo a modificar:** \src/components/RelatedContent/scoreNotes.ts\ (função \scoreRelatedNotes\)  
**Arquivo secundário:** \src/App.tsx\ (linha ~1048 para passar \calendarNotebookId\ ao componente)

**Impacto:** Nenhum para outros notebooks. Related content continuará funcionando normalmente.

## [2] Embed de Notas na Agenda

### Infraestrutura Existente
O mecanismo de embed **já existe e funciona**:

**Componentes:**
- \LinkedNoteBlock.ts\ — Node customizado do TipTap para armazenar referência de nota via \data-note-id\
- \LinkedNoteBlockView.tsx\ — View que renderiza:
  - Cabeçalho com título da nota, botão de expandir/colapsar, botão de navegar e botão de deslinkar
  - Body com conteúdo renderizado (quando expandido)
  - Carrega a nota em tempo real via \storage.getNote(noteId)\

**Como funciona atualmente:**
\\\
1. Editor tem um atalho para inserir linked note (likely: Ctrl+K ou similar)
2. Abre NotePicker com dropdown de filtro
3. Seleciona nota → insere <linked-note data-note-id="..."> no conteúdo
4. LinkedNoteBlockView renderiza e mantém sincronizado
\\\

### Funcionalidade Necessária para Agenda
**Status:** 95% pronto. Apenas precisa de:
1. Passar \llNotes\ (ou equivalente) ao Editor do Calendar notebook
2. Ativar o NotePicker **também** para notas de agenda

### Detalhes de Implementação

**Localização do NotePicker:**
- \src/components/Editor/NotePicker.tsx\ (linhas 12-79)
- Aceita array de notas filtradas por título
- Retorna \Note\ selecionada

**Localização do trigger:**
- Provavelmente em \src/components/Editor/Editor.tsx\ (linha ~350, atalho de teclado ou UI)
- Atualmente filtra \
otebookNotes\ — precisa expandir para "todas as notas" ou ter um seletor

**Caso de Uso Problema:**
\\\
User edita nota de agenda (03/06/2026)
- CurrentNote.notebookId = calendarNotebookId
- notebookNotes passa apenas notas do Calendar (vide App.tsx, linhas 426-431)
- Não consegue vincular notas de outros notebooks
\\\

**Solução (Médio):**
1. Em \App.tsx\, quando \ctiveNotebookId === calendarNotebookId\, passar **todas as notas** ao Editor
2. Ou criar um modo especial no Editor: "embed mode" que aceita qualquer nota

### Verificação de Loop Visual
- **LinkedNoteBlockView** suporta colapso (\collapsed\ attr)
- Renderiza conteúdo sob demanda
- **Não há proteção contra ciclos** (A links B, B links A)
- **Recomendação:** Adicionar validação no \NotePicker\ para alertar se a nota já contém um embed da nota atual (baixo esforço, opcional)

### Impacto
- Não afeta schema do banco (usa campo \content\ existente)
- Notas deletadas renderizam como "Linked note" com conteúdo vazio (graceful fallback já existe)

## [3] RAG nas Notas de Agenda

### Infraestrutura RAG Atual

**Sistema de RAG:** Totalmente funcional e integrado

**Fluxo:**
1. **Entrada:** User executa comando (ex: \/expand\, \/docs\) na nota
2. **Processamento:** \src/App.tsx\, linhas 536-599
   - Recupera \
otebookId\ da nota ativa
   - Obtém \isibleDocIds\ do notebook via \getNotebookVisibleDocumentIds(notebookId)\
   - Embeds o query via \embedText()\ (Tauri invoke + Xenova/transformers)
   - Busca chunks similares via \documentsSearchByIds(visibleDocIds, embedding, topK)\
   - Formata resultado em \agContext\ e passa ao modelo
3. **Saída:** Sistema insere RAG context no user message antes do LLM

**Gate atual:** RAG **ativado apenas se:**
- \isibleDocIds.length > 0\ (notebook tem documentos configurados)
- AND \!isDocsOnly\ OR \agContext\ existe (para \/docs\, é obrigatório)

### Status do RAG na Agenda
**Problema:** O RAG funciona com base em \
otebookId\, mas a **agenda (Calendar notebook) não é modelada como "um notebook com documentos"**.

**Hoje:**
\\\
Note on Calendar (notebookId = calendarNotebookId)
  ↓
getNotebookVisibleDocumentIds(calendarNotebookId)
  ↓
Returns: [] (vazio, porque ninguém configura docs no Calendar)
  ↓
RAG context = undefined
  ↓
Comando executado sem RAG
\\\

### Solução (Baixo a Médio)

**Opção A (Simples):** Adicionar um comando específico \/ask-with-rag\ que:
- Força RAG search em uma "knowledge base global" em vez de \
otebookId\
- Ou permite ao user especificar quais notebooks usar como fonte

**Opção B (Recomendada):** Permitir que usuário configure documentos visíveis **por nota**, não por notebook.

\\\
Para notas de agenda, adicionar:
1. Um painel simples no Editor (similar ao atual "Documents" do notebook)
2. Permite select de quais documentos/notebooks usar como fonte RAG
3. Salva em \
ote.metadata\ ou similar (requer schema change MÍNIMA)
\\\

**Opção C (Status Quo):** Não habilitar RAG para agenda; usuário cria notas normais (fora do Calendar) se quer RAG.

### Detalhes da Opção B (Recomendada)

**Modificações necessárias:**

1. **Schema** (mínimo):
   - Adicionar campo opcional a \Note\: \agDocumentIds?: string[]\ (JSON array)
   - No \storage/tauri.ts\, adicionar coluna \ag_doc_ids TEXT\ (nullable)

2. **Lógica** (em \App.tsx\, linhas 536-599):
   \\\	ypescript
   let visibleDocIds: string[] = []
   if (activeNote?.ragDocumentIds?.length) {
     visibleDocIds = activeNote.ragDocumentIds  // Usar override de nota
   } else if (notebookId) {
     visibleDocIds = await getNotebookVisibleDocumentIds(notebookId)  // Fallback
   }
   \\\

3. **UI** (novo componente ou painel):
   - Adicionar checkbox/dropdown ao Editor similar ao "Documents" do chat
   - Listar documentos globais disponíveis
   - Salvar seleção em \updateActive({ ragDocumentIds: [...] })\

**Impacto:**
- Baixo impacto geral; funciona em paralelo com sistema existente
- Notas sem \agDocumentIds\ comportam-se como hoje (fallback a \
otebookId\)
- Chat não é afetado (usa \olderDocIds\, que é independent)
