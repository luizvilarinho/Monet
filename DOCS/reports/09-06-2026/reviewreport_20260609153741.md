## Achados por severidade

### [importante] `applyingRemoteRef` pode ficar preso em `true` se um update remoto não gerar re-render
`src/hooks/useChat.ts:801-803` (release effect) + `:768-794` (handler `storage`)

A guarda anti-eco liga `applyingRemoteRef.current = true` no início do handler `storage` e a libera num `useEffect` sem deps que roda "por último no commit". O mecanismo depende de que **sempre haja um commit (re-render)** após o handler. Isso é verdade no caminho normal porque `loadConversations()/loadFolders()/loadLooseOrder()` retornam arrays com nova referência, então os setters agendam render.

Porém, se em algum caminho os setters resultarem em estado que o React considera igual (bail-out) e nenhum outro estado mudar, o release effect não roda e a flag fica `true` **permanentemente** — a partir daí TODOS os save-effects locais (`saveConversations`/`saveFolders`/`saveLooseOrder`) passam a ser pulados e escritas locais deixam de ser persistidas silenciosamente. Hoje isso não acontece porque as funções `load*` sempre criam referências novas, mas o invariante é frágil e não está protegido.

Recomendação: liberar a flag de forma determinística dentro do próprio handler (ex.: `queueMicrotask(() => { applyingRemoteRef.current = false })`) OU manter o release effect mas garantir explicitamente que ele sempre corre. Dado que o anti-loop real já é garantido pela comparação de conteúdo nos save-helpers (`saveConversations`/`saveFolders`/`saveLooseOrder` só gravam se a string mudou), avaliar se `applyingRemoteRef` ainda é necessário para conversations/folders/looseOrder — a comparação de conteúdo já quebra o eco. A flag só agrega valor onde a comparação de conteúdo não cobre (e nesses pontos o conteúdo, por definição, mudou). Simplificar reduz o risco.

### [importante] Inconsistência: `setFolderVisibleDocuments` grava no `localStorage` dentro do updater de estado e ignora a guarda anti-eco
`src/hooks/useChat.ts:1026-1037`

`setFolderVisibleDocuments` chama `saveFolders(next)` **dentro** do callback de `setFolders((prev) => { ... saveFolders(next); return next })`. Dois problemas:
1. É efeito colateral (I/O) dentro de um updater de estado — impuro, e em StrictMode o updater roda duas vezes (a `saveFolders` tem comparação de conteúdo, então a 2ª é no-op; ainda assim é padrão a evitar).
2. Bypassa a guarda `applyingRemoteRef` e o save-effect normal de `folders` (`:738-741`), divergindo de `setFolderSystemPrompt` (`:1008-1024`), que só atualiza estado e deixa o save-effect persistir.

Não é introduzido por este diff (pré-existente), mas é exercitado diretamente pela nova feature (botão "Documents" do `AssistantPanel`). Recomendação: remover o `saveFolders(next)` interno e deixar o save-effect de `folders` persistir, alinhando com `setFolderSystemPrompt`.

### [nit] Identidade da pasta `assistant` por NOME é colidível pelo usuário
`src/hooks/useChat.ts:439` (`ASSISTANT_FOLDER_NAME = 'assistant'`), usado em `dedupeAssistantFolders`, materialização (`:1297`), `ensureAssistantFolder` (`:888`) e resolução em `send` (`:1511`).

Toda a feature identifica a pasta do assistant por `f.name === 'assistant'`. Se o usuário criar/renomear uma pasta para "assistant" na janela principal, ela passa a ser tratada como a pasta do assistant (e pode ser consolidada/mesclada por `dedupeAssistantFolders`, perdendo `conversationIds`/config). É pré-launch e improvável, mas é um acoplamento por string frágil. Opcional: marcar a pasta com uma flag estável (ex.: `isAssistant: true` no `ChatFolder`) ou um id reservado, em vez de identidade por nome. Fica a critério do Orquestrador (mudança de schema do folder em localStorage — pré-launch permite).

### [nit] `mergeRemoteConversations` protege apenas a conversa em streaming desta janela, não o inverso completo
`src/hooks/useChat.ts:672-685`

A proteção preserva localmente a conversa que ESTA janela está gerando (`activeStreamRef.current?.convId`). Correto para o BUG 2 descrito. Observação: se as DUAS janelas estiverem em streaming na MESMA conversa simultaneamente (cenário não suportado/raro — o assistant cria sempre conversa nova), haveria disputa. Não é um caso real do produto (assistant sempre materializa conversa nova; o usuário não streama a mesma conversa em duas janelas), então é só registro — não exige ação.

### [nit] `createFolder` não é desestruturado no `AssistantPanel`, embora o plano o previsse
`src/components/AssistantPanel/AssistantPanel.tsx:32-50`

O plano (Etapa 4) mencionava destruturar `createFolder`, mas a implementação final usa `ensureAssistantFolder` (mais adequado, pois cria a pasta por nome sem conversa). `createFolder` não é necessário aqui. Sem ação — apenas confirma que o desvio do plano é correto e não deixou import/var órfã.

## Avaliação por categoria (consolidada)

1. **Correção** — Cumpre os requisitos:
   - Close-to-tray: `CloseRequested` na `main` faz `prevent_close()` + `hide()` (`lib.rs:1217-1224`); só na `main` (guard por `get_webview_window("main")`). Quit → `app.exit(0)` encerra tudo (`lib.rs:1199`). Clique esquerdo na tray e item Open mostram+focam a main (`lib.rs:1196-1213`).
   - Ctrl+M global via `tauri-plugin-global-shortcut`; falha de registro só loga (`lib.rs:1244-1252`), não crasha. Toggle correto: visível+focada → hide; senão show+focus+emit `assistant-shown` só ao mostrar (`lib.rs:24-39`).
   - Janela `assistant` declarada com `visible:false`/`alwaysOnTop`/`skipTaskbar`/`decorations:false` (`tauri.conf.json`). `main.tsx` roteia por `label === 'assistant'`.
   - Persistência preguiçosa: rascunho só-memória (`draftConversation`), materializado na 1ª mensagem na pasta `assistant` (`useChat.ts:1289-1315`); não cria conversa vazia; novo chat a cada Ctrl+M via `assistant-shown`.
   - System prompt + RAG da pasta `assistant` resolvidos por nome no turno da 1ª msg (`useChat.ts:1510-1513`, `:1532`).
   - Sync cross-window via `storage` com anti-loop por comparação de conteúdo nos save-helpers; `activeId` per-janela (ramo `ACTIVE_ID_KEY` removido do listener; save no-op quando `isAssistant`) — corrige BUG 1; `mergeRemoteConversations` + `applyingRemoteRef` corrigem BUG 2 (truncamento).
   - `dedupeAssistantFolders` consolida pastas duplicadas em todos os pontos de entrada (init via `reconcileOrders`, ramo `CONVERSATIONS_KEY` e `FOLDERS_KEY` do listener); idempotente (retorna a mesma ref com ≤1 pasta).

2. **Simplicidade** — Em geral boa. O par "comparação de conteúdo nos save-helpers" + "`applyingRemoteRef`" é redundante para conversations/folders/looseOrder (ver achado importante #1); poderia simplificar.

3. **Efeitos colaterais** — O listener `storage` agora roda também na janela principal (benéfico e desejado pelo plano). `dedupeAssistantFolders` reescreve `folders` no primeiro load mesmo na main — consolidação permanente de pastas `assistant` legadas; intencional. Não altera comportamento de pastas não-assistant. Cleanup do listener `storage` presente (`:793`). Cleanup do listener `assistant-shown` presente com guard `cancelled` (`AssistantPanel.tsx:106-109`). Sem leaks de listener observados.

4. **Padrões** — Segue as convenções do projeto (helpers `load*/save*` no topo, setters via `useCallback`, refs `*Ref.current = ...`, comentários em PT). Rust segue o estilo do `lib.rs` (builders, `let _ =` para descartar `Result`). A exceção é `setFolderVisibleDocuments` (achado importante #2) — divergente de `setFolderSystemPrompt`, mas pré-existente.

5. **Segurança** — Sem exposição de dados sensíveis. Capabilities adicionadas são mínimas e coerentes com o escopo autorizado (tray, global-shortcut, window show/hide/focus/start-dragging) e aplicadas a `["main","assistant"]`. Links no `AssistantBubble` só abrem `http(s)` via `openUrl` (`AssistantPanel.tsx:340-347`). `dangerouslySetInnerHTML` usa `renderMarkdown` (mesmo pipeline do chat principal — sanitização herdada). Sem loosening de CSP.

6. **Organização** — `AssistantPanel` em pasta própria com CSS module, reaproveitando `useChat`, `ModelSelector`, `ChatToolsMenu`, `FolderSystemPromptModal`, `FolderDocumentSelectorModal` — sem fork do chat engine, conforme exigido. Estrutura coerente com o restante do projeto.

## Notas sobre os pontos de foco solicitados

- **Concorrência cross-window**: a combinação anti-loop (comparação de conteúdo) + anti-eco (`applyingRemoteRef`) + proteção de streaming (`mergeRemoteConversations` por `activeStreamRef.convId`) cobre os cenários descritos (inclusão/deleção em tempo real, não-ressurreição de deletadas, fim do streaming com a main aberta, Ctrl+M sempre abrindo rascunho vazio). O único risco real é o `applyingRemoteRef` preso (achado importante #1), mitigado na prática pelas referências sempre-novas dos `load*`. Sem races de duplicação/perda de mensagem identificados no fluxo normal (assistant streama sempre uma conversa nova materializada; a main não ecoa cópia atrasada).
- **Persistência preguiçosa**: correta — rascunho nunca toca localStorage até a 1ª mensagem; materialização vincula à pasta `assistant`; `activeConversation`/guard de `activeId` consideram o rascunho (`:723-731`, `:1783-1788`); novo chat a cada Ctrl+M preservado.
- **dedupeAssistantFolders**: consolidação correta (une `conversationIds` sem repetir, preserva `systemPrompt`/`visibleDocumentIds` priorizando a pasta configurada, mantém posição da primária, idempotente) e plugada em todos os pontos de entrada de estado.
- **Lado Rust**: toggle, CloseRequested→hide só na main, Quit encerrando tudo, falha de global-shortcut tolerada — todos corretos. Sem vazamento/erro no setup. Ordem de `.plugin(global_shortcut)` após `.setup()` é válida em Tauri v2 (plugins inicializam antes do setup independentemente da posição no chain); `cargo check` confirma.
- **Dead code / inconsistências**: `startAssistantConversation` antigo foi substituído (sem resíduo); `ensureAssistantFolder` legado (que gravava direto no localStorage) foi removido nas iterações anteriores. Sem imports órfãos. Única inconsistência de padrão é `setFolderVisibleDocuments` (pré-existente, achado #2).
