## 3. `src/components/ChatPanel/ChatSidebar.tsx`

Mudança de maior superfície: estado `editingConvId`/`editingConvName`/`convEditingInputRef`, handlers `startConvEdit`/`commitConvEdit`/`cancelConvEdit`, `useEffect` de auto-foco, `onDoubleClick` no root do `SortableConversation` com guard, input inline, ações escondidas durante edição, e prop drilling de 7 novos props por `SortableFolder` → `FolderBody` → `SortableConversation`.

### Correção
- Espelha o padrão de renomeação de pasta (linhas 217-223, 239-265, 652-686). Verificado por leitura: o input da pasta tem exatamente a mesma estrutura de `onBlur`/`onKeyDown`/`onClick`/`onPointerDown`/placeholder/aria-label.
- Guard do `onDoubleClick` no root do `SortableConversation`:
  ```jsx
  onDoubleClick={(e) => {
    if ((e.target as HTMLElement).closest('button, input')) return
    if (editing) return
    onStartEdit(conv)
  }}
  ```
  - Bloqueia duplo-clique em botões de ação (delete, remover da pasta) e no próprio input. Correto.
  - Bloqueia re-entrada se já está em modo edição. Correto.
  - Não interfere com o dnd-kit: o `PointerSensor` usa `activationConstraint: { distance: 5 }` (linha 206), então duplo-clique estático não dispara drag.
- `onClick` e `onPointerDown` no input param propagação, impedindo `onSelect` (seleção) e drag de iniciar ao interagir com o input. Correto.
- `e.stopPropagation()` no `onKeyDown` do input impede que Enter/Escape disparem o `onKeyDown` do parent (que faz Enter/Space → `onSelect`). Correto.
- **Nit:** `e.stopPropagation()` no `onKeyDown` do input é chamado **incondicionalmente** (mesmo para teclas não tratadas como backspace/arrow). Consistente com o input de pasta (linha 680), que faz o mesmo. Sem impacto funcional, mas levemente redundante.
- Cross-type guard: `startConvEdit` cancela edição de pasta (`if (editingFolderId) cancelFolderEdit()`), e o `onStartEdit` de pasta (linha 486) cancela edição de conversa (`if (editingConvId) cancelConvEdit()`). Garante "apenas 1 edição por vez" cross-type. Vai além do literal do spec, justificado no execution report. Aceitável.
- Cobertura: input renderizado tanto para conv loose quanto in-folder (via `SortableConversation` props). Uma edição por vez (id é único).

### Simplicidade
- **Nit:** Prop drilling de 7 props novos (`editingConvId`, `editingConvName`, `onEditingConvNameChange`, `convEditingInputRef`, `onCommitConvEdit`, `onCancelConvEdit`, `onStartConvEdit`) por `SortableFolder` e `FolderBody`, mais 7 props no `SortableConversation` (que aliás **duplica** os 7 do folder mais os 5 do folder rename). A duplicação `onCommitEdit`/`onCancelEdit` (folder) vs `onCommitConvEdit`/`onCancelConvEdit` (conv) coexiste no mesmo componente `SortableConversation`, o que torna a interface de props dele densa.
  - Consistente com o padrão de folder rename existente, então não é regressão. Refator para um hook `useInlineEdit` ou context seria uma melhoria válida, mas é trabalho fora do escopo desta task.
- **Nit:** O `convEditingInputRef` é compartilhado entre o input de conv loose e o de conv in-folder. Como apenas um input é montado por vez (uma edição por vez), funciona. Mas é uma dependência implícita que não fica óbvia ao ler o código.

### Efeitos colaterais
- Nenhum.
- Estado `editingConvId` poderia ficar stale se a conv em edição fosse deletada externamente (não há UI que permita isso durante edição, pois os botões de ação ficam escondidos). Cenário teórico, sem impacto prático.

### Padrões
- Espelha fielmente o padrão de pasta. Boa consistência.

### Segurança
- Sem vulnerabilidades. Título é texto puro, renderizado pelo React.

### Organização e boas práticas
- Tipagem: `React.RefObject<HTMLInputElement | null>` no prop type é compatível com `useRef<HTMLInputElement>(null)` em React 19. Build passou.
- Separação de responsabilidades: handlers ficam no `ChatSidebar` (top-level), `SortableConversation` é apresentacional. Correto.
- Sem subscriptions vazadas.
