# Task: Renomear conversas no Chat Sidebar

## Objective
Permitir que o usuário atribua um nome custom a cada conversa listada no `ChatSidebar` (seja "loose" ou dentro de pasta), para facilitar localização. Hoje o título exibido é apenas o `deriveTitle` truncado a partir da 1ª mensagem do user (50 chars), o que torna conversas com perguntas parecidas ("o que é X?", "explique Y") quase indistinguíveis na lista. A renomeação precisa sobrescrever o título auto-derivado e persistir no localStorage junto com o resto da conversa.

## Context
- **Stack:** React 18 + TypeScript, Tauri 2 desktop, CSS Modules. Sem dependências novas necessárias.
- **Hook de estado:** `src/hooks/useChat.ts` — expõe `conversations`, `selectConversation`, `newConversation`, `deleteConversation`, operações de pasta, etc. Persiste em `localStorage` via `saveConversations()`. **Não tem hoje** nenhuma função `renameConversation` / `updateTitle`.
- **Container:** `src/components/ChatPanel/ChatPanel.tsx` consome `useChat` e repassa callbacks para o sidebar.
- **Sidebar:** `src/components/ChatPanel/ChatSidebar.tsx` (650+ linhas) renderiza pastas + conversas loose. O componente `SortableConversation` (linha ~410) é o que renderiza cada linha de conversa, hoje com `<span>{conv.title}</span>` + botões "remover da pasta" e "deletar".
- **Padrão de referência:** renomeação de pastas já existe no mesmo arquivo via duplo-clique no header → `<input>` com Enter/Esc/blur, estado `editingFolderId`/`editingFolderName` no `ChatSidebar`. O design da nova feature deve **espelhar exatamente** esse padrão, só que aplicado a conversas.
- **Modelo de dados:** `ChatConversation` em `useChat.ts` já tem `title: string` — não precisa mexer no shape.

## Requirements

### Requisito 1 — Entrar em modo de renomeação por duplo-clique
- Duplo-clique no título da conversa (no span que mostra `conv.title`, em `SortableConversation`) entra em modo edição.
- O duplo-clique **não pode disparar** quando o alvo é um botão interno (delete, remover da pasta) ou já está em modo edição.
- Duplo-clique não pode conflitar com a seleção por clique único: ao sair do modo edição, o clique único continua selecionando a conversa normalmente.

### Requisito 2 — UI inline de edição
- Substituir o `<span>` do título por um `<input type="text">` durante a edição, com:
  - Valor inicial = `conv.title` atual
  - Foco automático e texto selecionado ao entrar no modo
  - Largura suficiente para digitar confortavelmente
  - Estilo consistente com o input de renomeação de pasta (`.folderRenameInput` em `ChatSidebar.module.css`) — criar `.convRenameInput` análogo

### Requisito 3 — Comportamento de commit/cancel
- **Enter** ou **blur** → commita o novo título
- **Escape** → cancela e mantém o título anterior
- Espelhar `commitFolderEdit` / `cancelFolderEdit` do `ChatSidebar` (linhas ~210)

### Requisito 4 — Validação
- Aplicar `trim()` antes de salvar
- Se o resultado do `trim()` for string vazia → **rejeitar** e manter o título anterior (não commita nada, não fecha o input ou exibe feedback visual)
- Aplicar `slice(0, 100)` — limite de **100 caracteres** no valor final persistido

### Requisito 5 — Persistência e re-derivação
- Adicionar função `renameConversation(id: string, title: string)` em `src/hooks/useChat.ts` que:
  - Atualiza `conv.title` no array `conversations`
  - Atualiza `conv.updatedAt` para `new Date().toISOString()`
  - Persiste via `saveConversations()` (já existente)
  - Retorna a conversa atualizada
- Expor `renameConversation` no retorno do hook
- **Não** chamar `deriveTitle()` em nenhum ponto — o título custom é "sticky": uma vez renomeado, futuras mensagens não devem sobrescrevê-lo. (Hoje o `deriveTitle` só roda na criação/migração, então isso já é o comportamento de fato; explicitar no comentário da função.)

### Requisitos 6 — Cobertura
- Funciona para conversas **loose** (fora de pasta) e para conversas **dentro de pasta**
- Funciona em **qualquer pasta**, independente do `systemPrompt` ou `visibleDocumentIds` da pasta
- Apenas **uma** conversa em modo edição por vez (ao entrar em edição numa, sair da anterior — espelhar a lógica de `editingFolderId`)

## Constraints
- **Reusar o padrão existente de renomeação de pasta** — não inventar UX nova. Mesmo handler de teclado, mesma classe base do input, mesma semântica de commit/cancel.
- **Não adicionar novos botões/ícones** — o trigger é só duplo-clique (decisão do Coordenador). Não colocar lápis no hover.
- **Não introduzir dependências novas** — só React + o que já está no projeto.
- **Não duplicar lógica de drag** — o duplo-clique precisa conviver com o `useSortable` (atributos/listeners) sem quebrar o drag. Provavelmente vai precisar parar propagação no `onDoubleClick` do título (similar ao que `onPointerDown` já faz nos botões).
- **Não usar modal** para renomear — manter inline, igual pasta.
- CSS via `ChatSidebar.module.css` (módulo existente), sem classes globais.

## Edge Cases
- **Título só com espaços** ("   ") → tratar como vazio, rejeitar
- **Título com 100+ caracteres** → truncar para 100 antes de salvar
- **Conversa atualmente sendo arrastada (DnD ativo)** → duplo-clique não deve disparar rename (PointerSensor tem `activationConstraint: { distance: 5 }`, então duplo-clique estático não vai disparar drag; mas o input também não pode abrir se houver drag em curso)
- **Duplo-clique num botão de ação (delete, remover da pasta)** → não disparar rename (já é tratado pelo guard do `onDoubleClick` no container, mas reforçar)
- **Clicar duas vezes em conversas diferentes rapidamente** → o segundo duplo-clique deve cancelar a edição da primeira e abrir na segunda (uma edição por vez)
- **Blur causado por clique em outro lugar do app** → commita o título válido (igual pasta)
- **Conversa com `title` legado > 100 chars** (de `deriveTitle` anterior) → ao exibir, mostrar o que tiver; ao entrar em edição, o input recebe o valor completo mas trunca ao salvar
- **Enter dispara enquanto a conversa está sendo editada em outro lugar** (não esperado, mas considerar) → já coberto pelo "uma edição por vez"
- **Cenário sem nenhuma mensagem na conversa** (`deriveTitle` retorna "New conversation") → renomeação funciona normalmente e sobrescreve
