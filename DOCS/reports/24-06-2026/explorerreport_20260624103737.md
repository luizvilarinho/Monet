# Explorer Report (2º round) — Notas/cadernos não persistem após reiniciar o app

## Contexto deste round
Coordenador confirmou: instala/abre sempre via Microsoft Store (MSIX) e fecha sempre clicando em "Sair"/Exit no ícone da bandeja — mesmo método nas duas ocasiões. As duas hipóteses do 1º round (divergência MSIX vs instalador direto; X da janela só minimizando) ficam descartadas. Este round usou evidência direta do disco do usuário (arquivos reais em `%APPDATA%`), não só leitura de código.

## Achado principal — evidência direta no disco

Inspecionei `C:\Users\User\AppData\Roaming\com.monet.app\` (o `app_config_dir()`/`app_data_dir()` real, sem virtualização MSIX neste momento — ver seção 3) e encontrei:

```
monet.db       946.176 bytes   LastWriteTime: 09/06/2026 10:40:47
monet.db-shm    32.768 bytes   LastWriteTime: 24/06/2026 10:16:35
monet.db-wal 4.169.472 bytes   LastWriteTime: 24/06/2026 10:16:37
```

**O arquivo principal `monet.db` não recebe escrita desde 09/06/2026 — há 15 dias.** Todas as escritas das últimas duas semanas (cadernos e notas criados nesse período, incluindo presumivelmente os do relato do bug) foram acumuladas só no arquivo WAL (`monet.db-wal`, ~4 MB), que nunca foi "checkpointed" de volta ao arquivo principal.

Isso explica o sintoma com precisão:
- **Durante a sessão**, o SQLite em modo WAL lê de forma transparente a combinação `monet.db` + `monet.db-wal` — por isso tudo funciona normalmente enquanto o processo está vivo (criar caderno, escrever nota, navegar entre telas).
- **Ao reabrir o processo**, se o WAL não for lido corretamente (ou for descartado/resetado por qualquer motivo antes da nova conexão abrir), o app volta a ver apenas o conteúdo "congelado" do `monet.db` de 09/06 — que pode já não ter o caderno/nota mais recente, dando a impressão de "os dados desapareceram".

Esse achado por si não prova ainda *por que* o checkpoint nunca conclui, mas é a causa raiz mais provável e concreta encontrada até agora — muda o diagnóstico de "divergência de path" (descartado) para **"WAL nunca é checkpointed/persistido no arquivo principal, e fica vulnerável a esse acúmulo crescer indefinidamente"**.

## 1. Handler de "Sair"/Exit na bandeja

`src-tauri/src/lib.rs:1211-1215`:
```rust
.on_menu_event(|app, event| match event.id().as_ref() {
    "open" => show_and_focus_window(app, "main"),
    "quit" => app.exit(0),
    _ => {}
})
```

`app.exit(0)` é a única ação do Quit — não há nenhum flush/close manual da conexão SQLite no código da aplicação antes dessa chamada.

Pela documentação oficial do Tauri 2 (`AppHandle::exit`): *"Exits the app by triggering RunEvent::ExitRequested and RunEvent::Exit"* — ou seja, em teoria `app.exit(0)` dispara corretamente os dois eventos do ciclo de vida, não é um kill abrupto equivalente a `std::process::exit()` direto.

O `tauri-plugin-sql` (`v2.4.0`, fonte em `tauri-plugin-sql-2.4.0/src/lib.rs:174-184`) registra exatamente esse handler:
```rust
.on_event(|app, event| {
    if let RunEvent::Exit = event {
        run_async_command(async move {
            let instances = &*app.state::<DbInstances>();
            let instances = instances.0.read().await;
            for value in instances.values() {
                value.close().await;   // fecha o pool sqlx -> deveria fazer checkpoint do WAL
            }
        });
    }
})
```
`run_async_command` é bloqueante (`block_on`), então em teoria o processo espera esse `close()` terminar antes de morrer de fato.

**Conclusão deste item:** no papel, o caminho de saída está corretamente instrumentado pelo plugin para fechar o pool no exit. Mas a evidência física (WAL de 15 dias não checkpointado) mostra que, na prática, isso não está garantindo a persistência. Hipóteses mais plausíveis para a divergência entre "teoria" e "evidência":
- O pool sqlx por padrão permite até 10 conexões simultâneas (não há `max_connections` customizado em `wrapper.rs`); se alguma conexão do pool ficar presa/किn-use no momento do `close()`, ou se o `close()` não aguardar de fato todas as conexões finalizarem o checkpoint dentro do tempo de vida do processo, o WAL pode não ser truncado.
- O processo pode estar sendo finalizado por um caminho diferente do que se espera — por exemplo, se o usuário também usa o botão "Open"/clique do tray para reabrir entre uma sessão e outra sem nunca de fato ter saído pelo Quit (ficando dias com o processo na bandeja, sem nunca disparar `RunEvent::Exit`), o checkpoint nunca seria forçado, e o WAL cresceria indefinidamente — isso é compatível com 15 dias de acúmulo.
- Falha silenciosa dentro do `close()`/checkpoint que não gera nenhum log visível (não há logging de erro nesse trecho do plugin, e o app não expõe esse tipo de falha na UI).

## 2. Rotina de inicialização/migração do banco

Revisei a lista completa de `Migration`s em `src-tauri/src/lib.rs:1027-1192` (registradas via `tauri_plugin_sql::Builder::default().add_migrations("sqlite:monet.db", migrations)`, linha 1287-1289):

- Versões 1 a 10. Todas usam `CREATE TABLE IF NOT EXISTS` / `ALTER TABLE ADD COLUMN` — idempotentes e seguras para notebooks/notes.
- A única `DELETE FROM`/`DROP TABLE` destrutiva é a migration v6 (linha 1124, `DELETE FROM documents;`) e afeta exclusivamente a tabela `documents` (Base de Conhecimento), não `notebooks`/`notes`.
- `tauri_plugin_sql`'s `Migrator` (sqlx) controla a versão aplicada via tabela própria (`_sqlx_migrations`) — migrations já aplicadas não re-executam. Não há lógica de versão "manual"/custom no código do app que possa disparar incorretamente.

**Conclusão:** nenhuma rotina de migração ou inicialização explica a perda — confirma o achado do 1º round.

## 3. Resolução de `app_data_dir()`/`app_config_dir()` sob MSIX

Achado relevante e não levantado no 1º round: **o `tauri-plugin-sql` não usa `app_data_dir()`, usa `app_config_dir()`** (`tauri-plugin-sql-2.4.0/src/wrapper.rs:79-82`):
```rust
let app_path = _app.path().app_config_dir().expect("No App config path was found!");
```
Enquanto `vec_db.rs:35` e `documents.rs:110` usam `app_data_dir()` diretamente. No Windows, isso **não causa divergência física**: confirmei no crate `dirs` (`dirs-6.0.0/src/win.rs:8,10`) que `config_dir()` e `data_dir()` chamam exatamente a mesma função, `dirs_sys::known_folder_roaming_app_data()` → `FOLDERID_RoamingAppData` (`%APPDATA%`). Ou seja, `app_config_dir()` e `app_data_dir()` resolvem para o mesmo caminho (`%APPDATA%\com.monet.app`) tanto para `monet.db` quanto para `monet-vec.db`/`documents`. Confirmado também no disco: ambos os arquivos (`monet.db` e `monet-vec.db`) estão na mesma pasta `C:\Users\User\AppData\Roaming\com.monet.app\`.

Sobre o empacotamento MSIX especificamente: o manifesto real gerado (`src-tauri/target/msix/stage/AppxManifest.xml`, idêntico ao `src-tauri/msix/AppxManifest.template.xml` do commit `0cbe96c`) declara o app como `Windows.FullTrustApplication` com capability `runFullTrust`, **sem** a capability `unvirtualizedResources` (que a própria documentação da Microsoft usa em templates de apps full-trust para *desabilitar* a virtualização). Isso teoricamente sujeitaria o processo à virtualização de pastas do Desktop Bridge (`FOLDERID_RoamingAppData`/`FOLDERID_LocalAppData` redirecionados para `%LOCALAPPDATA%\Packages\<PackageFamilyName>\RoamingState` / `LocalCache`).

**Porém, a inspeção direta do disco do usuário mostra que isso não está acontecendo na prática**: busquei recursivamente por `*.db` em `C:\Users\User\AppData\Local\Packages\LuizVilarinho.Monetnotes_55xygb3fecfnw\` (todas as subpastas: `AC`, `AppData`, `LocalCache`, `LocalState`, `RoamingState`, `Settings`, `SystemAppData`, `TempState`) e não encontrei nenhum arquivo de banco — o `monet.db` real e ativo está em `%APPDATA%\com.monet.app\` (caminho não-virtualizado). Não há indício de path duplicado/divergente entre execuções neste ambiente.

**Conclusão deste item:** path do banco é estável e único (`%APPDATA%\com.monet.app\monet.db`) nesta máquina, sem virtualização MSIX ativa de fato. Esse item NÃO é a causa raiz.

## 4. Outras hipóteses MSIX/Store (não confirmadas pelo código, mas plausíveis)

Como pedido, hipóteses adicionais específicas de MSIX/Store que podem contribuir ou ser causa alternativa, sem confirmação 100% no código:

- **Atualização automática do pacote pela Store**: se a Store atualizar o MSIX em segundo plano entre uma sessão e outra, o Windows pode finalizar o processo antigo de forma abrupta (sem garantir `RunEvent::Exit`) para liberar o pacote para atualização — isso bypassaria o `close()` do plugin-sql e deixaria o WAL pendente exatamente como observado. Compatível com o padrão "funciona, depois nas próximas vezes os dados mais recentes não aparecem".
- **Antivírus/Windows Defender ou indexador de busca** com um handle aberto no arquivo `monet.db`/`monet.db-wal` no momento do `close()`, impedindo o SQLite de obter o lock exclusivo necessário para fazer checkpoint completo (`TRUNCATE`) — silenciosamente faz o checkpoint falhar ou ser parcial, sem erro visível para o usuário.
- **App ficar dias na bandeja sem nunca disparar Quit de fato**: se o usuário entende "fechar o app" como clicar no X (que só esconde, ainda que ele tenha dito que usa Quit) intercaladamente com sessões onde de fato usa Quit, o WAL acumularia por several dias até o exit "real" acontecer — consistente com os 15 dias observados. Vale confirmar quão frequentemente o Quit é realmente usado vs. quanto tempo o processo fica ocioso na bandeja.
- **Mais de uma instância do processo rodando simultaneamente** (não há plugin de single-instance registrado em `lib.rs` — confirmado por busca, nenhum resultado para `single_instance`): se o usuário, por engano, abrir o app duas vezes (ex.: clique duplo no ícone da Store enquanto uma instância já está na bandeja), duas pools SQLite distintas (processos diferentes) disputam o mesmo arquivo WAL, o que pode interferir no checkpoint e gerar inconsistências de leitura entre instâncias.

## Causa raiz mais provável (resumo)

**O WAL do SQLite (`monet.db-wal`) não está sendo checkpointed/persistido de volta ao arquivo principal `monet.db` de forma confiável entre sessões**, apesar do `tauri-plugin-sql` registrar corretamente um handler em `RunEvent::Exit` que deveria fechar o pool e forçar esse checkpoint (`tauri-plugin-sql-2.4.0/src/lib.rs:174-184`), e do `app.exit(0)` do tray Quit (`src-tauri/src/lib.rs:1213`) disparar esse evento segundo a documentação oficial do Tauri.

Evidência direta: `monet.db` parado em 09/06/2026, `monet.db-wal` de 4 MB atualizado até hoje (24/06/2026), ambos em `C:\Users\User\AppData\Roaming\com.monet.app\`.

Isso desloca o diagnóstico de "bug de lógica no código de save" (já descartado nos dois rounds) e de "divergência de path MSIX" (descartado neste round com evidência direta de disco) para uma **falha intermitente no fechamento gracioso da conexão SQLite no exit** — seja por timing entre `app.exit()` e o pool sqlx, concorrência de múltiplas conexões no pool default, interferência externa (AV/indexador), ou o processo morrendo por um caminho que não passa por `RunEvent::Exit` (kill pela Store durante auto-update, ou outro encerramento fora do controle do código da aplicação).

## Nível de dificuldade (correção, etapa futura)
**Média.** Não é uma correção trivial de uma linha, pois a causa não está 100% isolada — mas há mitigação direta e de baixo risco: forçar `PRAGMA wal_checkpoint(TRUNCATE)` explicitamente (via comando customizado, não dependendo só do `close()` do pool) em pontos seguros (após cada save crítico, e/ou no evento de exit antes do plugin fechar o pool), e/ou configurar `journal_mode=DELETE` em vez de WAL para esse banco específico (elimina o problema de checkpoint, ao custo de performance/concorrência ligeiramente menor, irrelevante para o volume de dados de notas pessoais). Validar também se há mais de uma instância do processo rodando.

## Recomendação de escopo
1. Antes de codar, recomendo instrumentar uma forma simples de capturar evidência adicional do usuário: expor no app (ex. tela de Settings/About) o tamanho atual do `monet.db-wal` e a data de modificação do `monet.db`, ou logar no exit se o checkpoint foi bem sucedido — isso confirmaria definitivamente se o padrão "WAL cresce, principal para" se repete antes de escrever a correção.
2. Correção de baixo risco e alto valor, independente de confirmação adicional: adicionar um comando/rotina que force `PRAGMA wal_checkpoint(TRUNCATE)` no `monet.db` logo após cada `saveNote`/`saveNotebook` (ou, mais simples, trocar para `journal_mode=DELETE`/`journal_mode=PERSIST` nessa conexão específica, removendo o WAL do caminho crítico). Como o app é de uso pessoal/local (sem concorrência multi-processo real esperada), abandonar o WAL em favor do journal padrão elimina esta classe de bug por completo.
3. Mantida a recomendação do 1º round: propagar erros de save para a UI (`useNotes.ts:30-32`, `useNotebooks.ts:30-32`, `useSubjects.ts:33`), o que ajudaria a detectar falhas de checkpoint/lock em tempo real no futuro.
