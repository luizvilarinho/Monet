## 1. `src/storage/tauri.ts` — troca de `journal_mode` para DELETE

```ts
private db(): Promise<Database> {
  if (!this.dbPromise) {
    this.dbPromise = Database.load(DB_URL).then(async (db) => {
      try {
        await db.execute('PRAGMA journal_mode=DELETE')
      } catch (err) {
        console.error('failed to set journal_mode=DELETE', err)
      }
      return db
    })
  }
  return this.dbPromise
}
```

**Correção**: correto. `storage` é singleton (`src/storage/index.ts:51`) e `TauriStorage` é o único ponto que chama `Database.load` (único call site no projeto) — logo só existe uma `dbPromise` em todo o app. Todas as chamadas a `db()` (de qualquer hook, em qualquer ordem de disparo no mount) aguardam essa mesma promise resolver, e o PRAGMA roda dentro do `.then()` que a constrói. Isso elimina estruturalmente a race condition que causou a regressão da 1ª rodada — não é uma questão de "qual hook chama primeiro", é garantido pelo cache de promise. Confirmei isso lendo `src/storage/index.ts` e todos os call sites de `getNotebooks`/`getNotes`/`getSubjects`.

O `try/catch` está correto: nunca rejeita o `dbPromise`, então uma falha do PRAGMA (ex.: lock por outro processo) deixa o banco em WAL mas o app continua funcionando — degradação aceitável, não crítica.

**Efeitos colaterais**: nenhum identificado. `journal_mode` é uma propriedade persistida no arquivo (não por conexão), então rodar o PRAGMA uma vez por sessão antes de qualquer outra query é suficiente, como documentado no comentário do código.

**Padrões**: o comentário no código (linhas 150-156) é claro e explica o "porquê" (lock exclusivo, ordem sequencial) — bom nível de documentação para uma decisão não-óbvia, evita que um futuro autor "simplifique" e reintroduza a regressão.

Nenhum problema encontrado neste arquivo.

## 2. `src-tauri/Cargo.toml` + `src-tauri/src/lib.rs` — `tauri-plugin-single-instance`

```rust
tauri::Builder::default()
    .plugin(tauri_plugin_single_instance::init(|app, _args, _cwd| {
        show_and_focus_window(app, "main");
    }))
    .manage(RequestRegistry(Mutex::new(HashMap::new())))
    .setup(|app| { ... })
```

**Correção**: correto. É o primeiro plugin registrado, como exige a documentação oficial do Tauri (precisa interceptar antes de qualquer outra inicialização). O callback reaproveita `show_and_focus_window` (`lib.rs:20-26`), que já trata corretamente janela minimizada/oculta na bandeja (`unminimize` + `show` + `set_focus`) — coerente com o fato de que a janela principal nunca é de fato destruída ao clicar no X (só esconde, via `prevent_close` em `lib.rs:1238-1242`).

Versões resolvidas em `Cargo.lock`: `tauri-plugin-single-instance 2.4.2` contra `tauri 2.10.3` — compatíveis (linha major `2`).

**Sobre a ausência de `#[cfg(desktop)]`**: concordo com a decisão do codeAgent de não adicionar o guard. Nenhum outro plugin do arquivo usa `cfg(desktop)`/`cfg(mobile)` (confirmado via grep no arquivo inteiro — zero ocorrências), não há `mobile_entry_point` nem configuração de targets Android/iOS no projeto. Adicionar o guard apenas neste plugin introduziria uma inconsistência sem benefício prático hoje. Se o projeto ganhar suporte mobile no futuro, o guard deve ser adicionado neste ponto — mas isso é fora do escopo atual.

**Efeitos colaterais**: nenhum. O handler de Quit (`app.exit(0)`, `lib.rs:1216`) e o fechamento do pool SQL pelo `tauri-plugin-sql` via `RunEvent::Exit` continuam funcionando como antes; o plugin novo só intercepta uma 2ª instância do processo, redirecionando para focar a janela existente em vez de abrir um segundo processo (que seria a fonte de duas conexões concorrentes ao mesmo arquivo SQLite — o cenário que esta correção existe para prevenir).

Nenhum problema encontrado neste arquivo.

## 3. `saveError` em `useNotes.ts`, `useNotebooks.ts`, `useSubjects.ts`, `App.tsx`, `App.css`

**Correção**: correto. Cada hook expõe `saveError: string | null`, setado nos `catch` de save/remove/(reorder, só em `useSubjects`) e limpo (`setSaveError(null)`) em caso de sucesso da mesma operação. `App.tsx` consome os três e renderiza um único banner com prioridade fixa `note > notebook > subject` (`noteSaveError || notebookSaveError || subjectSaveError`).

**Padrões**: o padrão `error: string | null` por hook + `<div role="alert">` já existe em pelo menos 5 componentes do projeto (`ChatPanel`, `AssistantPanel`, `AiPanel`, `SaveToNoteModal`, `KnowledgeBaseModal`), confirmado por grep. A nova implementação segue exatamente essa convenção, incluindo a ausência de botão de dismiss (o `errorRow` de `ChatPanel.tsx:400-404` também não tem um) — não é uma lacuna introduzida pela mudança, é consistente com o que já existe.

`App.css` é importado por `App.tsx` como CSS global (`import './App.css'`, não CSS module) — confirmado lendo o import existente no próprio arquivo. Logo adicionar `.saveErrorBanner` em `App.css` (em vez de um `.module.css`) é o padrão correto para este arquivo especificamente, mesmo que outros componentes do projeto usem CSS modules. `var(--accent-warn)` sem fallback é seguro aqui porque a variável é definida em `:root` no próprio `App.css` (linha 12), carregado globalmente — diferente dos `.module.css` de outros componentes que usam fallback (`var(--accent-warn, #ef9f27)`) por precaução de import isolado, mas que não é necessário neste arquivo.

**Duplicação entre os 3 hooks**: existe (cada hook repete `useState` + 2x `setSaveError` por operação), mas é do mesmo tamanho/natureza da duplicação já presente entre os hooks para `notes`/`loaded`/`save`/`remove`/`create` — os 3 hooks já não compartilham abstração alguma hoje. Introduzir um hook genérico de erro só para esta mudança seria inconsistente com o resto do arquivo e desproporcional ao tamanho do problema (3 linhas por hook). `[nit]`, não bloqueante.

**Comportamento com múltiplas falhas simultâneas**: o banner mostra só o primeiro erro da prioridade fixa; os outros ficam "escondidos" até o primeiro ser limpo por uma operação bem-sucedida do mesmo hook (não há dismiss manual). Documentado como decisão consciente no executionreport ("falhas de disco devem ser raras"). Aceitável para o caso de uso (falha de I/O local, não erro de rede recorrente) e consistente com o padrão de banner único já usado no projeto. `[nit]`.

**Dead code**: não encontrei nenhum. O executionreport menciona que a instrumentação de debug temporária (`'DEBUG: ' + ...`) foi removida de `useNotebooks.ts` na 2ª rodada — confirmei lendo o arquivo atual: não há vestígio dela.

Nenhum problema bloqueante encontrado nestes arquivos.
