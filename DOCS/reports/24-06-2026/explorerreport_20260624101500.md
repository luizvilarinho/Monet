# Explorer Report — Notas/cadernos não persistem após reiniciar o app

## Descrição do problema

Usuário cria caderno → cria nota → escreve. Navegar para o chat e voltar mantém os dados (estado em memória OK). Fechar o app inteiro e reabrir: caderno e nota desaparecem, como se nunca tivessem existido.

## 1. Onde fica o código de criar/salvar cadernos e notas

**Frontend:**
- `src/hooks/useNotebooks.ts:18-33` (`save`) e `:44-57` (`create`) — cria/atualiza `Notebook` em estado React e chama `storage.saveNotebook`.
- `src/hooks/useNotes.ts:18-33` (`save`) e `:44-61` (`create`) — idem para `Note`, chama `storage.saveNote`.
- `src/storage/index.ts:43-51` — `createStorage()` decide em runtime entre `TauriStorage` (desktop) e `BrowserStorage` (web), verificando `window.__TAURI__`/`__TAURI_INTERNALS__`.
- `src/storage/tauri.ts:144-269` — `TauriStorage.saveNotebook` (linha 160) e `TauriStorage.saveNote` (linha 244) executam `INSERT ... ON CONFLICT(id) DO UPDATE` via `@tauri-apps/plugin-sql`, contra `DB_URL = 'sqlite:monet.db'` (linha 13).
- Disparo do save de nota: `src/App.tsx:632-635` (`updateActive`), chamado pelo `Editor` a cada `onChange` (sem debounce — ver item 2).

**Backend/Tauri (Rust):**
- `src-tauri/src/lib.rs:1027-1192` — lista de `Migration`s registradas no `tauri_plugin_sql::Builder` (linha 1287-1289), incluindo `CREATE TABLE notebooks`/`notes` (migration v1) e alterações incrementais (v8 `date`, v9 `subject_id`, etc.).
- O plugin-sql é só uma camada sqlx; não há comando Tauri customizado para notebooks/notes — toda a escrita/leitura é feita pelo frontend via `Database.load('sqlite:monet.db')` (`tauri.ts:148`).

## 2. Quando o save é disparado, e se persiste de fato em disco

- **Sem debounce.** Busquei por `debounce`/`setTimeout` ligado a save em todo `src/` e não há nenhum. O `Editor` chama `onChangeRef.current(md)` a cada alteração de conteúdo (`src/components/Editor/Editor.tsx:368`), que sobe direto para `onChange={(content) => updateActive({ content })}` em `App.tsx:994` → `updateActive` (`App.tsx:632-635`) → `saveNote` (síncrono, `await`).
- O `save` de `useNotes`/`useNotebooks` primeiro atualiza o estado React (otimista) e **em seguida** faz `await storage.saveNote(note)` / `await storage.saveNotebook(nb)` — uma escrita real no SQLite (`INSERT ... ON CONFLICT DO UPDATE`), não apenas estado em memória.
- `tauri-plugin-sql` v2 (`@tauri-apps/plugin-sql@^2.4.0`, `tauri-plugin-sql = "2"` no `Cargo.toml:23`) faz autocommit por `execute()` por padrão (sem transação aberta) — cada keystroke gera um commit imediato no arquivo `monet.db`. Isso foi corroborado pelo padrão idêntico usado em `subjects` (`useSubjects.ts`) e pelo uso de `journal_mode = WAL` nas conexões Rust diretas (`documents.rs:129`, `vec_db.rs:46`), o que é consistente com escrita committada a cada operação.
- **Ponto de atenção (não a causa raiz, mas mascara falhas):** os catches de erro em `useNotes.ts:30-32`, `useNotebooks.ts:30-32` e `useSubjects.ts:33` apenas fazem `console.error(...)` — nunca propagam o erro para a UI. Como o estado React já foi atualizado *antes* do `try`, se a escrita em disco falhar (permissão, disco cheio, lock), a UI continua mostrando a nota/caderno como se estivesse salvo, sem qualquer aviso ao usuário. Isso por si não filtra o que aconteceu nesse bug, mas é uma lacuna real que dificultaria diagnosticar o problema relatado a partir da experiência do usuário (nenhum erro visível).

## 3. Onde fica o load na inicialização, e se lê do mesmo arquivo do save

- `useNotebooks.ts:10-16` e `useNotes.ts:10-16` chamam `storage.getNotebooks()` / `storage.getNotes()` dentro de `useEffect` no mount do componente raiz (`App.tsx:169-176`).
- `TauriStorage.getNotebooks` (`tauri.ts:152-158`) e `TauriStorage.getNotes` (`tauri.ts:227-233`) fazem `SELECT * FROM notebooks/notes ORDER BY updated_at DESC` na mesma instância `Database.load('sqlite:monet.db')` (cacheada em `this.dbPromise`, `tauri.ts:145-150`).
- **Conclusão: save e load usam exatamente a mesma string de conexão (`sqlite:monet.db`) e a mesma API (`@tauri-apps/plugin-sql`)** — não há dois adaptadores diferentes em jogo nem dois arquivos distintos no código do app. O path físico do arquivo (`<app_data_dir>/monet.db`) é resolvido pelo plugin a partir do `identifier` do Tauri em runtime, igual para save e load porque é o mesmo processo/sessão.
- Validação cruzada: o backend Rust (`documents.rs:107-114`, `vec_db.rs:32-39`) resolve `app_data_dir()` da mesma forma (`app.path().app_data_dir()`) para abrir conexões diretas a `monet.db`/`monet-vec.db`, confirmando que todo o app converge no mesmo diretório base por execução.

## 4. Divergência de path entre dev/produção ou outra causa de "path mismatch"

- **Identifier dev vs produção é real e documentado**: `src-tauri/tauri.conf.json:5` define `"identifier": "com.monet.app"` (produção); `src-tauri/tauri.conf.dev.json:4` sobrescreve para `"com.monet.app.dev"`. Como o Tauri deriva `app_data_dir()` do identifier, dev grava em `%APPDATA%\com.monet.app.dev\` e produção em `%APPDATA%\com.monet.app\` — **pastas e arquivos `monet.db` completamente diferentes e isolados**. Isso está documentado na memória do projeto (`project-dev-prod-db-separation`, criada em 2026-06-17) como já resolvido/intencional, mas é a forma mais simples de "ver dados em uma sessão e não na próxima": basta abrir o app por um caminho em uma sessão (ex.: `npm run tauri:dev`) e pelo executável instalado/MSIX em outra.
- **Build MSIX não introduz um identifier extra**: o pacote MSIX documentado em `project-microsoft-store-msix.md` empacota o `target/release/monet.exe` já compilado a partir do `tauri.conf.json` de produção padrão — usa o mesmo `com.monet.app`. Não encontrei um terceiro `tauri.conf*.json` nem script de build MSIX no estado atual do repo (`src-tauri/msix/` não existe nesta árvore de trabalho), então não há evidência de um identifier divergente *dentro* da distribuição de produção (EXE/MSI vs MSIX).
- **Fechar a janela (X) não mata o processo**: `src-tauri/src/lib.rs:1232-1240` registra `on_window_event` na janela `main` que, em `WindowEvent::CloseRequested`, chama `api.prevent_close()` e `win.hide()` — ou seja, clicar no X apenas oculta a janela para a bandeja; o processo (e a conexão SQLite) continuam vivos. O único caminho que de fato mata o processo é o item "Quit" do tray (`lib.rs:1213`, `app.exit(0)`) ou matar via Task Manager/encerrar sessão do Windows. Isso é relevante para reproduzir o bug: "fechar o app inteiro" pelo relato do usuário provavelmente significa usar o Quit do tray (ou reiniciar a máquina), não o X da janela — vale confirmar com o usuário qual ação ele está realizando, porque isso direciona se o processo morre de forma limpa (commit já em disco, sem risco) ou abrupta.
- **Sem evidência de redirecionamento de path por sandboxing MSIX que cause perda**: apps com identidade de pacote MSIX (mesmo `Windows.FullTrustApplication`) recebem virtualização de `%APPDATA%`/`%LOCALAPPDATA%` para uma pasta isolada por pacote (`%LOCALAPPDATA%\Packages\<PackageFamilyName>\...`), mas essa virtualização é estável entre execuções do mesmo pacote instalado — só mudaria se o Package Family Name mudasse (reinstalação com identidade diferente, ou trocar entre instalar via MSIX/Store e instalar via instalador EXE/MSI direto, que NÃO tem identidade de pacote e usa o `%APPDATA%` real). **Esse é um candidato forte**: se o usuário tem o app instalado tanto pela Microsoft Store (MSIX, com identidade de pacote → AppData virtualizado) quanto baixou o instalador `.exe`/`.msi` direto do GitHub Releases (sem identidade de pacote → AppData real), são dois `app_data_dir()` físicos diferentes no disco mesmo com o mesmo `identifier` lógico — alternar entre os dois fechando um e abrindo outro reproduziria exatamente o sintoma relatado (dados "desaparecem" porque na verdade está olhando para outro arquivo `monet.db`).
- **Migrations são idempotentes e não destrutivas para notebooks/notes**: revisei toda a lista de `Migration` em `lib.rs:1027-1192` — apenas a migration v6 (`DELETE FROM documents`) é destrutiva, e afeta somente a tabela `documents` (Base de Conhecimento), não `notebooks`/`notes`. Migrations usam `CREATE TABLE IF NOT EXISTS` e `ALTER TABLE ADD COLUMN`, seguras para re-execução.
- **Nenhuma rotina de limpeza/reset automática toca notebooks/notes**: busquei `DROP TABLE`/`DELETE FROM`/`remove_dir`/`remove_file` em todo `src-tauri/src` — os únicos achados relevantes a startup são migrations (já tratado) e `cleanup_stuck_indexing` (`documents.rs:1340-1361`), que só faz `UPDATE documents SET status='error' WHERE status='indexing'` — não apaga notas/cadernos.

## 5. Causa raiz mais provável (com evidência)

O código de persistência de notebooks/notes em si está correto: o save é síncrono, sem debounce, e commita de fato no SQLite a cada alteração (`src/hooks/useNotes.ts:18-33`, `src/storage/tauri.ts:244-269`), e o load na inicialização lê exatamente do mesmo arquivo/conexão (`src/storage/tauri.ts:148, 227-233`). Não encontrei nenhum bug de lógica (race condition, transação sem commit, save apenas em memória) que explique perda de dados dentro de uma mesma forma de execução do app.

A hipótese mais provável, dado o padrão "funciona na sessão, zera ao reabrir o processo", é **divergência do diretório físico de dados (`app_data_dir()`) entre duas formas diferentes de abrir o app**, não um bug de código de save/load. Duas variantes concretas, em ordem de probabilidade:

1. **Instalação MSIX (Microsoft Store) vs instalador EXE/MSI direto** — ambos usam o mesmo `identifier` lógico (`com.monet.app`, `tauri.conf.json:5`), mas o MSIX roda com identidade de pacote Windows, que redireciona `%APPDATA%` para uma pasta virtualizada por pacote, fisicamente diferente do `%APPDATA%\com.monet.app\` usado pelo instalador direto. Se o usuário alternar entre as duas formas de instalação/execução (por exemplo: testou a versão da Store, depois abriu uma cópia development/direta, ou vice-versa), cada uma "vê" seu próprio `monet.db` vazio. Evidência: `project-microsoft-store-msix.md` (memória do projeto) confirma que o MSIX reusa o build padrão sem identifier próprio, e o comportamento de AppData virtualization é um fato conhecido de empacotamento MSIX no Windows (não código deste repo, mas highly relevante à branch `ms-store` atual).
2. **Execução via `npm run tauri:dev` vs app instalado em produção** — `tauri.conf.dev.json:4` força `identifier: com.monet.app.dev`, gravando em `%APPDATA%\com.monet.app.dev\`, totalmente isolado do `%APPDATA%\com.monet.app\` de produção (`project-dev-prod-db-separation`, memória de 2026-06-17). Menos provável neste caso porque o relato soa como uso do app já empacotado, não do ambiente de desenvolvimento, mas vale eliminar perguntando ao usuário como ele abriu o app nas duas ocasiões.

**Recomendação para a próxima etapa (não implementada aqui):** antes de tocar em código, confirmar com o usuário exatamente *como* ele fechou e reabriu o app (Quit do tray vs X da janela vs reiniciar o PC) e *qual* atalho/instalação ele usa (ícone da Store vs `.exe` instalado manualmente vs atalho de dev). Se for confirmado o cenário de instalação dupla (Store + direto), a correção é de produto/distribuição (orientar o usuário a manter uma única forma de instalação, ou — como mudança de código — adicionar uma verificação ao primeiro start que avise se outro `app_data_dir` irmão com dados existe). Se for confirmado uso exclusivo de uma única instalação, a investigação precisa voltar ao código com logging adicional (ex.: expor ao usuário o path resolvido de `app_data_dir()` em Settings) para capturar evidência direta na máquina do usuário — o código revisado aqui não contém um bug determinístico que explique perda de dados commitados.

## Nível de dificuldade (correção, etapa futura)
- Se a causa for confirmada como divergência de path entre instalações: **Baixa** (mudança de orientação/diagnóstico, não de lógica de persistência).
- Se a causa for outra (ainda não identificada em código): **Média**, pois exigiria instrumentação adicional (logging do path resolvido, captura de erros de save expostos à UI) para reproduzir no ambiente do usuário antes de poder corrigir.

## Recomendação de escopo
Sugiro, como próximo passo antes do `plannerAgent`/`codeAgent`, uma pergunta direta ao usuário (via Coordenador) sobre como ele abre o app nas duas ocasiões (ícone usado, se instalou por mais de uma via) — a resposta decide se o trabalho restante é uma correção de código (pouco provável, dado o que foi encontrado) ou uma orientação de produto/instalação. Adicionalmente, recomendo — independentemente da causa raiz aqui — corrigir a lacuna de error handling silencioso (`useNotes.ts:30-32`, `useNotebooks.ts:30-32`, `useSubjects.ts:33`) para que falhas de save fiquem visíveis ao usuário, já que isso dificulta diagnosticar este tipo de bug a partir de relatos futuros.
