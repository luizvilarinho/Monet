## src/hooks/useChat.ts

1. **Correção** — Implementação corresponde ao plano em todos os pontos: campos `memory`/`memoryEnabled` em `ChatFolder`, normalização em `normalizeFolder`, defaults em `makeNewFolder`, merge em `dedupeAssistantFolders`, injeção de `MEMORY_SYSTEM_PROMPT` + conteúdo da memória sempre que `memoryEnabled` (inclusive com `systemPromptMode === 'replace'`, confirmado lendo o bloco em `send()` — a injeção ocorre fora do `if/else` que monta `systemMessages` a partir do prompt da pasta), payload de tools migrado para array aditivo (`toolDefs`) com `update_folder_memory` desacoplado da checagem de Tavily, e handler de tool call reescrito para iterar `p.toolCalls` (consistente com a mudança de payload do backend). Sem bugs óbvios.
2. **Simplicidade** — Direto, sem abstrações desnecessárias. O loop `for (const tc of p.toolCalls)` com `toolSearchMessage` declarado fora do loop é a solução mais simples dado que `web_search`/`deep_research` continuam mutuamente exclusivas (só um dos dois pode aparecer no array).
3. **Efeitos colaterais** — Nenhum efeito colateral fora do escopo identificado. O array de dependências do `useCallback` de `send()` não inclui `setFolderMemory`/`setFolderMemoryEnabled`, mas ambos são derivados de `setFolders` (estável), igual ao padrão já usado por `setFolderSystemPrompt` — não é um problema real, já documentado como tal no Execution Report.
4. **Padrões** — Segue exatamente o padrão de `systemPrompt`/`setFolderSystemPrompt` já existente (setters, normalização, merge). Comentários em PT-BR delimitando blocos, igual ao resto do arquivo.
5. **Segurança** — Sem exposição de dados sensíveis. Conteúdo da memória é texto livre gerado pelo próprio modelo com o conteúdo da conversa do usuário, mesma superfície de risco que já existe para `systemPrompt`/histórico de chat (localStorage, sem sanitização adicional — comportamento pré-existente, não introduzido aqui).
6. **Organização e boas práticas** — Tipagem adequada (`parsedArgs` com `memory?: unknown`, checado com `typeof === 'string'` antes de usar). Nenhuma lógica de negócio vazando para fora do hook.

`[nit]` A instrução `MEMORY_SYSTEM_PROMPT` (texto enxuto fornecido pelo Coordenador) fala em "keep it concise" e "don't be afraid to make mistakes", mas não dá nenhuma orientação sobre quando **não** atualizar a memória (o texto anterior, mais longo, tinha os critérios de novidade/durabilidade/não generalizar reação pontual, incluindo o exemplo de "nojo de lagarta"). Isso é uma troca deliberada e já validada pelos testes do Coordenador, então não é reportado como desvio — só registro de que o texto atual é mais permissivo quanto à frequência de update, o que é consistente com o motivo da troca (o texto longo fazia o modelo gravar memória raramente demais).

## src-tauri/src/lib.rs

1. **Correção** — O fix do bug de parsing de múltiplas tool calls está correto: substitui as duas variáveis flat (`tool_call_name`/`tool_call_args`) por `BTreeMap<u64, (String, String)>` indexado pelo campo `index` do delta SSE, iterando `tool_calls.as_array()` em vez de só `.get(0)`. A ordem de emissão (`tool_calls.values()`, BTreeMap ordenado por chave ascendente) preserva a ordem dos índices originais. O filtro `!name.is_empty()` no `.collect()` final evita emitir entradas sem nome de tool.
2. **Simplicidade** — Abordagem mínima e direta para o problema (não introduz complexidade desnecessária; `BTreeMap` é a escolha certa para "poucos itens, ordenados por índice numérico pequeno").
3. **Efeitos colaterais** — Mudança de contrato do evento `openrouter://tool_call` (`tool_name`/`arguments_json` → `tool_calls: Vec<ToolCallItem>`) é breaking, mas coordenada corretamente com o consumidor (`src/lib/openrouter.ts` e `src/hooks/useChat.ts` foram atualizados no mesmo commit/sessão). Não há outro listener desse evento no projeto (não verificado exaustivamente fora do escopo, mas os três arquivos alterados formam o par produtor/consumidor completo).
4. **Padrões** — Consistente com o estilo já usado no arquivo (structs `#[derive(Clone, Serialize)]` com `rename_all = "camelCase"`, `.and_then()` chains para parsing de JSON opcional).
5. **Segurança** — Sem exposição de dados sensíveis nova.
6. **Organização** — OK.

`[nit]` Condição de emissão `finish_reason == Some("tool_calls") && !tool_calls.is_empty()` verifica o mapa não vazio, não se **há alguma entrada com nome não-vazio**. Em tese, se todos os deltas de tool_calls chegassem sem nome (só `index` + `arguments`, sem `function.name` em nenhum chunk — cenário atípico mas não impossível dependendo do provider), o `filter` no `.collect()` produziria `items: []` e o evento seria emitido com `tool_calls: []`, causando um round-trip extra vazio no frontend (o loop `for (const tc of p.toolCals)` simplesmente não itera nada, e o fluxo segue para `toolApiMessages` sem nenhum tool de fato executado). Isso é uma variação marginal do comportamento anterior (que simplesmente não emitia nada nesse caso) — efeito prático mínimo, mas vale registrar para o caso de resultar em uma chamada de API duplicada silenciosa caso o cenário aconteça na prática.

1. **Correção** — Paridade implementada: ícone `Brain` na titlebar, modal, toggle consciente de `assistantFolder`. `ensureAssistantFolder()` adicionado ao mount e ao listener de `assistant-shown`, com `startAssistantConversation`/`ensureAssistantFolder` ambos `useCallback` com identidade estável (`[]` e `[setFolders]` respectivamente, confirmado lendo as definições em `useChat.ts`) — não há risco de loop no `useEffect` com esse array de dependências.

3. **Efeitos colaterais** — `[importante]` Mudança de comportamento não trivial: antes desta task, a pasta `assistant` só era criada (persistida em localStorage) quando o usuário clicava em "System prompt"/"Documents" ou enviava a 1ª mensagem na janela Ctrl+M. Como `ChatSidebar.tsx`/`ChatPanel.tsx` não filtram pastas por nome (confirmado — nenhuma ocorrência de `ASSISTANT_FOLDER_NAME` em `ChatSidebar.tsx`), isso significa que uma pasta "assistant" vazia (sem conversas) passa a aparecer na sidebar da janela principal assim que o usuário abrir o Ctrl+M pela primeira vez, mesmo sem nunca ter enviado mensagem por lá. Esse comportamento é **intencional e já justificado no Plan Report** (Etapa 11, item 4: sem isso, o toggle de memória ficaria oculto até o usuário interagir com algo) — mas é uma mudança de comportamento visível para o usuário que vale a pena o Coordenador confirmar que é aceitável, já que não estava explicitamente coberta pelo Task Report original (que falava de paridade de *memória*, não de timing de criação de pasta). Não bloqueia o merge — é uma consequência colateral pequena e reversível, mas registro como `[importante]` por ser um comportamento novo, visível ao usuário, fora do que o Task Report descreveu literalmente.
4. **Padrões** — Botão `Brain` segue exatamente o padrão dos botões `Sliders`/`Files` já existentes (mesmo `styles.iconBtn`, mesmo padrão `onClick` chamando `ensureAssistantFolder()` antes de abrir o modal).
