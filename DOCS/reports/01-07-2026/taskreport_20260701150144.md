# Task: Memória de Pasta (Folder Memory) no chat

## Objective
Adicionar ao chat do Monet uma memória textual por pasta (`ChatFolder`), gerenciada de forma autônoma pelo próprio modelo via tool call, para dar continuidade de contexto entre conversas na mesma pasta sem exigir que o usuário reexplique contexto ou que o modelo releia o histórico completo.

## Context
- Documento fonte do brainstorm: `DOCS/features/memoria.md`.
- "Pasta" = `ChatFolder` (agrupamento de conversas do chat), definido em `src/hooks/useChat.ts`, persistido em localStorage (`FOLDERS_KEY = 'monet:chat-folders'`) — não é o "notebook"/caderno de notas.
- Precedente arquitetural direto: `ChatFolder.systemPrompt` + `systemPromptMode` (`'replace'|'append'`), editado via `FolderSystemPromptModal.tsx`, acionado por ícone na linha da pasta em `ChatSidebar.tsx` (padrão a replicar para o ícone/modal de memória).
- Montagem do system prompt final ocorre em `send()` (`src/hooks/useChat.ts`, ~linha 1485): `CHAT_SYSTEM_PROMPT` (padrão do app) + `folderPrompt` (replace ou append) + data atual + RAG system message + search system message, nessa ordem, antes de virar `apiMessages`.
- Tool calling é client-driven: backend Rust (`src-tauri/src/lib.rs`) só repassa `tools`/`tool_choice` e emite evento `openrouter://tool_call` ao detectar `finish_reason == "tool_calls"`; o frontend (`toolCallHandlerRef.current` em `useChat.ts`) executa a tool localmente e reinicia o streaming. Hoje só uma tool é enviada por chamada (`if/else` entre `web_search` e `deep_research`).
- Painel "Ferramentas" (`ChatToolsMenu.tsx`) hoje é estado **global** do app (`ChatTools` em localStorage, `TOOLS_KEY`), não por pasta — precisa virar consciente da pasta ativa para o novo toggle.
- Já existe um indicador visual reaproveitável para o toast: `webSearchActive` em `ChatPanel.tsx` (linhas 453-458) renderiza um banner verde (`.webSearchProgress`/`.webSearchDot`, `ChatPanel.module.css` linhas 664-685, cor `var(--accent-ai, #5dcaa5)`) com dot pulsante acima do composer, texto "Searching the web...", `role="status"`. Hoje sua visibilidade é amarrada à duração real da busca (liga/desliga em `useChat.ts`), não a um timer fixo — para a memória (escrita síncrona, sem round-trip externo), o comportamento de "aparece e some sozinho após alguns segundos" deve ser implementado via `setTimeout`, reaproveitando o mesmo estilo visual, só trocando o texto (ex.: "Folder memory updated").
- Ícones do app usam `@phosphor-icons/react` (ex.: `FileText`, `SidebarSimple` em `ChatSidebar.tsx`) — não introduzir nova lib de ícones; o ícone novo de memória deve vir dessa lib (ou seguir o padrão de SVG inline já usado por `SystemPromptIcon`, que é o precedente direto do botão a replicar).
- Escopo confirmado com o Coordenador: a feature se aplica **somente ao chat** (`ChatFolder`), não a `/comandos` do editor de notas nem a outras superfícies de IA do app. Por ser um campo do `ChatFolder`, aplica-se a qualquer pasta de chat da mesma forma que `systemPrompt` já se aplica hoje (inclusive à pasta especial do assistente global/Ctrl+M, sem tratamento especial).
- Decisões confirmadas com o Coordenador durante esta análise:
  - Persistência da memória: **localStorage**, seguindo o mesmo padrão de `systemPrompt` em `ChatFolder` (não criar tabela SQLite nova).
  - Tool calling: o modelo pode receber **várias tools ao mesmo tempo** em geral. A única exclusividade que se mantém é a já existente entre `web_search` e `deep_research` (nunca as duas juntas, por risco de conflito de funcionamento) — `update_folder_memory` deve poder ser oferecida junto com qualquer uma das duas (ou com nenhuma), exigindo o payload de tool única virar array condicional.

## Requirements
- Novo campo no `ChatFolder`: `memoryEnabled: boolean` (flag por pasta) e `memory: string` (conteúdo textual livre da memória), com setter análogo a `setFolderSystemPrompt`.
- Novo ícone na linha da pasta em `ChatSidebar.tsx` (mesmo padrão visual/comportamental de `SystemPromptIcon`), abrindo um modal (`FolderMemoryModal`, análogo a `FolderSystemPromptModal`) com o texto completo da memória, editável livremente (ler, editar, apagar trechos, apagar tudo).
- Novo toggle "Memória da pasta" no box de Ferramentas (`ChatToolsMenu.tsx`), abaixo do toggle de Web Search, mas com escopo por pasta (lê/escreve em `ChatFolder.memoryEnabled` da pasta ativa, não no `ChatTools` global) — requer tornar `ChatToolsMenu` consciente da pasta atual.
- Nova constante de prompt `MEMORY_SYSTEM_PROMPT`, independente do `CHAT_SYSTEM_PROMPT` e do `systemPrompt` opcional da pasta, contendo os critérios de julgamento descritos na seção 2 do documento fonte (teste de durabilidade, novidade, reuso, tema sustentado, intenção declarada, saliência cognitiva/emocional, não generalizar reação pontual) e 2-3 exemplos concretos (incluindo o caso "nojo de lagarta" como exemplo do que não registrar).
- Composição do prompt final deve injetar, quando `memoryEnabled` estiver true, **sempre**: `MEMORY_SYSTEM_PROMPT` + conteúdo atual de `ChatFolder.memory`, como mensagens `system` adicionais — independente do `systemPromptMode` da pasta ser `'replace'` (hoje esse modo descarta `CHAT_SYSTEM_PROMPT` inteiramente; a injeção de memória não pode seguir essa mesma exclusão).
- Nova tool `update_folder_memory`, oferecida ao modelo somente quando `memoryEnabled` estiver true (e o modelo suportar tools), recebendo o conteúdo atual da memória como contexto e permitindo reescrever o blob inteiro (não append). Deve poder coexistir no mesmo payload com `web_search` OU `deep_research` (quando ativas) — `web_search` e `deep_research` continuam mutuamente exclusivas entre si (comportamento existente mantido).
- Handler de tool call novo em `toolCallHandlerRef.current` para `update_folder_memory`: ao ser chamado com sucesso, persiste o novo texto em `ChatFolder.memory` (localStorage) e dispara o toast de feedback.
- Toast de feedback: reaproveitar o padrão visual/estrutural já existente em `ChatPanel.tsx` (`webSearchActive` → `.webSearchProgress`/`.webSearchDot`, banner verde com dot pulsante acima do composer, `role="status"`), com texto próprio (ex.: "Folder memory updated") e visibilidade controlada por `setTimeout` (alguns segundos, auto-dismiss), já que a escrita da memória é local/síncrona e não tem uma fase "em andamento" real para amarrar o indicador. Clique opcional abrindo o modal de memória fica como melhoria futura, não obrigatório na v1.
- Modelo usado para decidir/escrever a memória: o mesmo modelo principal da conversa (sem roteamento para modelo mais barato nesta v1).

## Constraints
- Sem schema fixo para o conteúdo da memória — texto livre, formato/organização a critério do próprio modelo por pasta.
- Não duplicar preferências de estilo de resposta na memória — isso já é coberto pelo `systemPrompt` opcional existente da pasta.
- Sem memória global de usuário (fora de pasta) nesta versão.
- Sem roteamento para modelo mais barato na extração/gestão de memória nesta versão (mencionado como otimização futura no documento fonte).
- Persistência via localStorage (decisão confirmada) — não criar tabela/migration SQLite para isso.
- `web_search` e `deep_research` continuam mutuamente exclusivas entre si (nunca oferecidas juntas) — essa regra não muda; a novidade é permitir `update_folder_memory` combinada com qualquer uma das duas.
- Ícone novo deve vir da lib de ícones já usada no projeto (`@phosphor-icons/react`) ou seguir o padrão de SVG inline do `SystemPromptIcon` — não adicionar nova dependência de ícones.
- Toast de feedback deve reaproveitar o componente/estilo visual já existente (`webSearchProgress`/`webSearchDot`), não criar um sistema de notificação novo do zero.
- Reutilizar os padrões já existentes (`ChatFolder`, `FolderSystemPromptModal`, ícones de `ChatSidebar`, composição de system prompt em `send()`) em vez de criar estruturas paralelas.
- Projeto em fase de pré-lançamento sem usuários ativos — breaking changes são aceitáveis se necessário, mas não é objetivo desta task.

## Edge Cases
- Pasta com `systemPromptMode === 'replace'` e `memoryEnabled === true`: `MEMORY_SYSTEM_PROMPT` + conteúdo da memória devem ser injetados mesmo assim (não podem depender do prompt padrão estar ativo).
- Modelo atual não suporta tools (`supportsTools === false`): não há fallback sensato para "escrita" de memória (diferente de web search, que tem fallback de busca prévia sem tool call) — a tool simplesmente não pode ser oferecida; documentar que a memória fica de fato inoperante (leitura de contexto ainda funciona, escrita não) nesses modelos.
- `web_search`/`deep_research` (uma das duas, nunca ambas) + `update_folder_memory` ativas ao mesmo tempo: o payload de tools passa a ser um array com múltiplas entradas; o handler de tool_call no frontend e o parsing de `tool_calls` no backend Rust (hoje assume uma única tool por `finish_reason`) precisam suportar múltiplas tools **disponíveis** simultaneamente, mesmo que o modelo só chame uma por turno.
- `dedupeAssistantFolders` (merge de pastas duplicadas) precisa incluir os novos campos `memory`/`memoryEnabled` na lógica de merge, mesmo padrão hoje aplicado a `systemPrompt`/`visibleDocumentIds`.
- Memória muito grande: a poda/resumo é responsabilidade do próprio modelo via instrução no `MEMORY_SYSTEM_PROMPT` (seção 2.5 do documento fonte) — não há enforcement automático de tamanho no código nesta v1.
- Tool chamada em turno onde não há mudança real relevante: maioria dos turnos não deve chamar a tool (comportamento esperado, não é bug).
- Toast dispara durante o streaming de uma resposta longa: não deve interromper nem bloquear o fluxo de escrita/leitura do usuário; se um novo toast disparar enquanto o anterior ainda está visível (dois turnos seguidos atualizando a memória), definir se substitui o timer atual ou enfileira — comportamento mais simples é reiniciar o timer com a mensagem mais recente.
- Toast reaproveitado (`webSearchProgress`) e o novo toast de memória podem, em tese, ficar ativos ao mesmo tempo (web search rodando + memória atualizada no mesmo turno) — garantir que os dois banners não se sobreponham visualmente (empilhar ou o de memória substituir o de busca quando a busca já terminou).
