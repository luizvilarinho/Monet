# Relatório de Análise — Prompt pessoal por pasta no Chat

**Data:** 2026-05-17  
**Solicitação:** Permitir que o usuário defina um prompt de sistema por pasta de chat.  
Exemplo: pasta "Biologia" → prompt "você é um professor de Biologia e está ensinando para um Mestrando na área".

---

## Viabilidade

**Viável.** Nenhum impedimento arquitetural ou técnico.

O prompt da pasta substitui completamente o prompt global (`CHAT_SYSTEM_PROMPT`) quando definido. Se a pasta não tiver prompt, usa-se o global normalmente.

---

## Dificuldade: **Baixa**

### Motivos

- O prompt de sistema já é injetado na posição correta (`useChat.ts:997`) como primeiro item do array de mensagens da API. A alteração é apenas condicionar qual string usar.
- O objeto `ChatFolder` já é serializado/deserializado para `localStorage` como JSON — qualquer campo novo é persistido automaticamente, sem migração de schema.
- Nenhuma alteração no backend Rust/Tauri, no SQLite, nem no storage adapter.
- A `ChatSidebar` já possui um bloco de ações por pasta (`folderActions`) onde um botão de editar prompt pode ser adicionado com padrão visual já existente.

---

## Arquivos a modificar

| Arquivo | O que muda |
|---|---|
| `src/hooks/useChat.ts` | 1. Campo `systemPrompt?: string` no `ChatFolder`<br>2. Default `''` no `makeNewFolder()`<br>3. Nova função `setFolderSystemPrompt(id, prompt)`<br>4. No `send()`, buscar a pasta da conversa ativa e usar seu prompt se existir |
| `src/components/ChatPanel/ChatSidebar.tsx` | 1. Prop `onSetFolderSystemPrompt` na `SortableFolder`<br>2. Novo botão (ícone de engrenagem) no `folderActions`<br>3. Modal ou textarea inline para editar o prompt |
| `src/components/ChatPanel/ChatSidebar.module.css` | Estilo do botão de prompt (reutilizar classes `folderActionBtn`) |

**Total estimado:** ~40 linhas de código novo/alterado.

---

## Pontos de atenção

1. **Segurança:** O prompt do usuário é injetado diretamente como `system` role no payload da API. Como o frontend chama o backend Rust que faz a requisição (chave nunca exposta ao frontend), não há risco de injeção de prompt malicioso além do que o próprio usuário digitar.

2. **UI:** Recomendo um modal simples com `<textarea>` ao invés de edição inline no header — prompts podem ser longos e não cabem no espaço lateral de 180px. O modal pode ser aberto por um botão de engrenagem ao lado dos botões `+` e `×` já existentes.

3. **Longevidade:** Se no futuro os chats migrarem de `localStorage` para SQLite, o campo `systemPrompt` migra junto naturalmente como coluna da tabela `chat_folders`.

4. **UX complementar sugerida:** Exibir um indicador visual sutil (ex: ícone ou cor diferente) no header da pasta quando ela tiver um prompt customizado definido, para o usuário saber quais pastas têm comportamento especial.

---

## Recomendação

A implementação é simples e de baixo risco. Pode ser feita em uma única sessão. A maior decisão de design — substituir vs anexar o prompt global — favorece a substituição, que dá mais controle ao usuário e mantém o código mais simples.
