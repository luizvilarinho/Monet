# Análise de Impacto: CodeMirror 6 → TipTap

**Data:** 2026-05-15  
**Objetivo:** Eliminar o botão "preview" e ter um editor WYSIWYG (o que você escreve já aparece formatado)  
**Nível de Dificuldade:** **ALTA**

---

## O problema real

O CodeMirror 6 é um editor de código. Ele exibe markdown como texto bruto (`**negrito**`, `## Título`), o que obriga o usuário a usar um botão "preview" separado para ver o resultado formatado. O TipTap é um editor WYSIWYG — negrito aparece negrito enquanto você digita, títulos aparecem grandes, checkboxes aparecem como caixas clicáveis. A motivação da troca é legítima e o objetivo final é alcançável.

---

## O que precisa ser reescrito

### 1. Editor.tsx inteiro (~1200 linhas) — **Risco Alto**
O arquivo é quase 100% CodeMirror 6. As principais peças que precisam ser portadas:

| Componente CM6 atual | Equivalente TipTap |
|---|---|
| `StateField` + `WidgetType` para checkboxes | `TaskList` + `TaskItem` (extensão nativa do TipTap) |
| `ToggleBlockWidget` (blocos de resposta IA colapsáveis) | Node View customizado (ProseMirror NodeView) |
| `createMarkerDecorations` (botões ↓inserir / ↑remover / ×) | Node View do mesmo bloco acima |
| `createCommandDecorations` (colorir linhas `/comando`) | Extension customizada com decoração ProseMirror |
| `commandCompletionSource` (autocomplete de `/comandos`) | `@tiptap/suggestion` (propósito exato desta extensão) |
| `formatting.ts` (bold, italic, listas, headings) | API de comandos do TipTap (`editor.chain().toggleBold()`) |
| `FormattingToolbar` (barra flutuante) | Suportada, mas requer refatoração da API |

### 2. Os marcadores HTML embutidos — **Risco Crítico**
Este é o maior risco. Atualmente os blocos de resposta da IA ficam armazenados no markdown bruto da nota assim:

```
/resumir <!--monet:abc123-->
<!--monet-embed:abc123-->
> **Resumo gerado pela IA**
> conteúdo...
<!--monet-embed-end:abc123-->
```

O TipTap trabalha com JSON (formato ProseMirror) internamente. Para manter o armazenamento em markdown é necessário usar `@tiptap/extension-markdown`, que aceita serializers/deserializers customizados. Os marcadores `<!--monet-embed:...-->` não são markdown padrão e precisarão de uma extensão de parsing dedicada. Se isso não for feito corretamente, todas as notas existentes com respostas embutidas podem ter o conteúdo corrompido na migração.

### 3. Formato de dados das notas existentes
As ~notas salvas no SQLite com blocos embutidos precisam de uma estratégia de migração. Migrar o schema ou manter compatibilidade retroativa são as duas opções, mas ambas têm custo.

---

## O que NÃO precisa ser reescrito

- `commandParser.ts` — funções puras, sem dependência de CM6. Pode ser reaproveitado quase integralmente.
- `useCommands.ts`, `useAi.ts`, `useNotes.ts` e demais hooks — sem CM6.
- Todo o backend Rust (Tauri, SQLite, RAG, PDF parsing).
- O sistema de /comandos em si — a lógica, os prompts, o streaming. Só a detecção no editor muda.
- O CSS de tema (variáveis CSS, paleta de cores).

---

## O que o TipTap resolve bem

| Problema atual | Como TipTap resolve |
|---|---|
| Preview separado necessário | WYSIWYG nativo — sem preview |
| Checkbox renderizado como texto `- [ ]` | `TaskList`/`TaskItem` embutidos |
| Negrito/itálico aparece como `**texto**` | Renderização inline em tempo real |
| Títulos aparecem como `## texto` | Renderização inline em tempo real |
| Autocomplete de /comandos é trabalhoso | `@tiptap/suggestion` resolve de forma mais limpa |

---

## Recomendações

**1. Viabilidade:** A troca é viável, mas é uma **reescrita do editor**, não uma refatoração. Estime 2–3 semanas de desenvolvimento focado.

**2. Ponto crítico a resolver antes de começar:** Definir como os blocos embutidos (`<!--monet-embed:...-->`) serão representados no TipTap. A melhor opção é criar um Node customizado `EmbedBlock` que serializa/deserializa de e para o formato de marcadores HTML atual. Isso preserva as notas existentes.

**3. Estratégia sugerida — migração em paralelo:**
   - Criar `EditorTipTap.tsx` ao lado do `Editor.tsx` atual
   - Implementar feature-by-feature: primeiro o editor base, depois checkboxes, depois /comandos, por último os embed blocks
   - Só substituir o Editor.tsx quando todas as funcionalidades estiverem validadas

**4. Dependências a adicionar:**
   - `@tiptap/react`, `@tiptap/starter-kit`
   - `@tiptap/extension-task-list`, `@tiptap/extension-task-item`
   - `@tiptap/suggestion` (para /comandos)
   - `@tiptap/extension-markdown` (para manter armazenamento em markdown)

**5. O que NÃO fazer:** Não tentar migrar incrementalmente o Editor.tsx atual. O CM6 e o TipTap têm APIs incompatíveis — misturar os dois não funciona. A abordagem paralela (arquivo novo) é mais segura.

## Resumo executivo

| | |
|---|---|
| **Vale a pena trocar?** | Sim — o TipTap é a ferramenta certa para o caso de uso |
| **Risco principal** | Blocos de resposta IA embutidos (marcadores HTML no markdown) |
| **Esforço estimado** | Alto (reescrita completa do editor) |
| **Notas existentes corrompem?** | Não, se o Node customizado for feito corretamente |
