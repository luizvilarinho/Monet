- **Feature 2** (Captura de tela): **Viável — Média-Alta dificuldade**

## Feature 1: Envio de Imagens para Modelos com Vision

### Viabilidade: ✅ VIÁVEL
### Dificuldade: MÉDIA

**Análise:**

OpenRouter suporta modelos com vision natively (Claude 3, GPT-4 Vision, Gemini). A API aceita content como array com múltiplos tipos (text, image_url).

**Alterações necessárias:**

1. **Tipo ChatMessage** — Estender content de string para string | ContentBlock[]
2. **Input do chat** — Adicionar file picker + preview de thumbnails
3. **Backend** — Adaptar openrouter_stream_messages() para aceitar content como JSON genérico
4. **Storage** — Migração de localStorage (imagens grandes → IndexedDB em fase 3)

**O que já existe:**
- Markdown renderer já renderiza imagens nas respostas (via emark-gfm)
- Editor já suporta drag-and-drop de imagens

**Impacto:**
- Afeta: useChat.ts, ChatPanel.tsx, types
- Não afeta: RAG, comandos /, persistência SQLite
- Estimativa: 2-3 dias

---

## Feature 2: Captura de Tela (Print Screen)

### Viabilidade: ✅ VIÁVEL
### Dificuldade: MÉDIA-ALTA

**Análise:**

Tauri 2 não possui plugin nativo de screenshot, mas existem opções comunitárias (tauri-plugin-screenshot) ou native Rust (Windows API).

**Fluxo proposto:**
1. Usuário ativa via hotkey (Ctrl+Shift+P) ou botão
2. Tauri executa command de captura
3. Exibe preview modal
4. Usuário confirma → anexa ao input
5. Envia via Feature 1

**Alterações necessárias:**

1. **Backend** — Novo Tauri command capture_screenshot()
2. **Frontend** — Botão + modal de preview
3. **Cargo.toml** — Adicionar plugin ou deps de screenshot
4. **Global shortcut** (opcional) — Registrar hotkey em tauri.conf.json

**Complexidade:**
- Simples: Tauri command é 4-5 linhas Rust
- Complexo: Multi-monitor support, DPI awareness, plugin choice

**Impacto:**
- Prerequisito: Feature 1 (file picker + anexo)
- Estimativa: 2-3 dias

---

## Feature 3: Exibição de Imagens da Web nas Respostas

### Viabilidade: ✅ VIÁVEL
### Dificuldade: BAIXA

**Análise:**

O markdown renderer já suporta completamente:
- Sintaxe ![alt](url) → renderiza via emark-gfm
- HTML <img> tags → whitelist permitida em ehype-sanitize

**Validação:** Se o modelo retornar markdown com imagens, são renderizadas corretamente.

**Alterações necessárias:**

1. **CSS** — Estilo para <img> no chat (responsive, padding, borders) — 10-15 linhas
2. **Error handling** (opcional) — Fallback para imagens quebradas

**Impacto:**
- Afeta: ChatPanel.module.css apenas
- Zero mudanças no backend ou types
- Estimativa: 2-3 horas

## Considerações Importantes

**Segurança:**
- Feature 1: Base64 em localStorage pode degradar performance (usar IndexedDB em fase 3)
- Feature 2: Screenshot local, sem auto-upload
- Feature 3: rehype-sanitize bloqueia scripts; CORS pode falhar em imagens

**Impacto no Schema:**
- Feature 1 é breaking change (refatoração de ChatMessage.content)
- Requer migração de histórico existente em localStorage

**Dependências:**
- Feature 1: Nenhuma biblioteca nova obrigatória
- Feature 2: tauri-plugin-screenshot (comunitário, mantido)
- Feature 3: Zero dependências novas
