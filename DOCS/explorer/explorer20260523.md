# Análise de Viabilidade: Refatoração do Sistema de RAG

**Data:** 23/05/2026  
**Explorador:** Claude Code Explorer  
**Tarefa:** Refatorar sistema de RAG de Cadernos para Base de Conhecimento global com seleção por contexto

---

## 1. Problema

Atualmente, o RAG funciona apenas no contexto de **Cadernos** (Notebooks):
- Documentos são anexados diretamente ao caderno (upload local)
- Os documentos indexados ficam visíveis apenas para aquele caderno específico
- Não há compartilhamento de conhecimento entre cadernos
- Não existe RAG no contexto de **Chat por Pastas**

A proposta é:
1. Criar uma **Base de Conhecimento global** gerenciada centralmente (upload, listagem, exclusão)
2. No **Caderno**, em vez de anexar docs, o usuário seleciona quais docs da base estão disponíveis
3. Em **Pastas de Chat**, o usuário também seleciona quais docs da base estão disponíveis para os chats daquela pasta

---

## 2. Arquitetura Atual

### Backend Rust (src-tauri)
- **Banco monet.db** (tauri-plugin-sql): Armazena metadados de documentos
  - Tabela `documents` com chaves estrangeiras para `notebook_id`
- **Banco monet-vec.db** (rusqlite + sqlite-vec): Armazena embeddings em formato vetorial
  - Tabela virtual `vec_chunks` (768 dimensões, OpenRouter embeddings)
  - Tabela `chunks_meta` com índices para `notebook_id` e `document_id`
- **Processamento**: Tauri commands: `documents_upload`, `documents_delete`, `documents_search`, `embed_text`
- **Indexação**: Assíncrona em background, emite eventos de status

### Frontend React (src/)
- **Hook**: `useDocuments(notebookId)` — carrega docs do caderno específico
- **Modal**: `DocumentsModal` — UI de upload/listagem/exclusão por caderno
- **RAG no Editor**: Quando `/comando` é disparado, `App.tsx` chama `documentsSearch(notebookId, embedding, topK)` para buscar chunks relevantes
- **Chat**: Estrutura em `useChat.ts` define `ChatFolder` (com `conversationIds`) mas RAG não está integrado no Chat

## 3. Mudanças Necessárias

### 3.1 Banco de Dados

**monet.db (tauri-plugin-sql)**

Tabela `documents` atual (vinculada a notebook):
```sql
CREATE TABLE documents (
  id TEXT PRIMARY KEY,
  notebook_id TEXT NOT NULL,
  name TEXT,
  status TEXT,
  ...
)
```

**Opção A (Recomendada):** Remover `notebook_id` de `documents`
- `documents` vira tabela global de armazenamento
- Criar tabelas de vinculação (muitos-para-muitos):
  - `notebook_document_visibility` (notebook_id, document_id, visible: bool)
  - `folder_document_visibility` (folder_id, document_id, visible: bool)

**Opção B:** Manter `documents` como está, criar nova tabela `global_documents`
- Mais disruptivo, mantém dois conceitos

**monet-vec.db (rusqlite)**
- `chunks_meta` atual filtra por `notebook_id` na busca
- Mudar para filtrar por lista de documentos visíveis para o caderno/pasta

### 3.2 Backend Rust

**Novos Tauri commands:**
- `documents_list_global()` — lista todas as docs da base, sem filtro de notebook
- `documents_upload_global(source_path)` — upload para base sem vincular a notebook
- `documents_delete_global(document_id)` — delete de doc global (impacta todas as vinculações)
- `documents_set_notebook_visibility(notebook_id, document_id, visible: bool)`
- `documents_set_folder_visibility(folder_id, document_id, visible: bool)`

**Modificar:**
- `documents_search(notebook_id, query_embedding, topK)` → passar lista de document_ids visíveis
- Ou: `documents_search_context(context_type: 'notebook'|'folder', context_id, query_embedding, topK)`

**Migrations:**
- Nova migration (v6) para reorganizar dados (Opção A recomendada)
- Backfill: Todos os docs existentes com notebook_id ganham vinculação automática em `notebook_document_visibility` com `visible=true`

### 3.3 Frontend React

**Componentes novos:**
- `KnowledgeBaseModal` — Modal global de gerenciamento da base (upload, listagem, exclusão, tags/categorização opcional)
- Botão na Toolbar ou Sidebar principal para abrir KnowledgeBaseModal

**Modificações:**
- `DocumentsModal` — passa a ser seletor de docs já criadas, com checkbox de visibilidade
- `ChatFolder` (em `useChat.ts`) — adicionar `visibleDocumentIds: string[]`
- `ChatPanel` — expor UI de seleção de docs para a pasta

**Hooks:**
- Novo `useKnowledgeBase()` — gerencia docs globais
- Estender `useDocuments()` — passar `visibility: Map<documentId, boolean>` para caderno/pasta

## 4. Análise de Impacto

### Impacto Alto:
1. **Estrutura de dados** (banco) — múltiplas tabelas novas, migration complexa
2. **Lógica de busca RAG** — filtro de contexto + visibilidade (pequeno, isolado)
3. **UI de seleção** — novo componente global + ajustes em modals existentes

### Impacto Médio:
1. **Chat com RAG** — adicionar suporte de RAG a pastas (nova feature, não quebra existente)
2. **Backfill de dados** — todos os docs atuais precisam ser vinculados

### Impacto Baixo:
1. **Tauri commands** — adicionar novos, manter compatibilidade dos antigos
2. **Embeddings** — sem mudança, reutiliza `sqlite-vec`

### Riscos:
- **Data loss em migration**: Backfill precisa garantir que nenhum documento desapareça
- **Performance**: Join com tabelas de visibilidade — índices em notebook_id, folder_id, document_id
- **UX confusa**: Diferenciar "documentos globais" vs "visibilidade por contexto" — precisa ser claro no design

## 5. Nível de Dificuldade

**MÉDIA**

### Motivo:
- Não é trivial (envolve DB, backend, frontend, RAG search)
- Mas é bem **isolado** — RAG é modular, banco tem tabelas claras
- Sem dependências de terceiros novos
- Fluxo de dados bem compreendido

### Estimativa:
- **DB + Migrations**: 2-3 dias (backend dev)
- **Backend Rust**: 1-2 dias (novos commands + busca contextual)
- **Frontend + Hooks**: 2-3 dias (nova modal, modificar modals, estender useChat)
- **Testes + QA**: 1-2 dias
- **Total**: 1-1.5 semanas (1 dev)

## 6. Recomendações

### Scope MVP:
1. Base de Conhecimento global (upload, listagem, exclusão)
2. Cadernos com seleção de visibilidade de docs
3. Manter RAG funcionando conforme hoje, mas filtrando por visibilidade

### P2:
- Pastas de Chat com suporte a RAG (investigar se `ChatFolder` usa DB ou localStorage)
- Categorização/tags para docs
- Busca/filtro na base por nome

### Pontos de Atenção:

**1. Naming claro na UI:**
- "Base de Conhecimento" = repositório global
- "Documentos disponíveis" = visibilidade para o caderno/pasta atual

**2. Migration segura:**
- Script de backfill com testes
- Backup da tabela `documents` antes da migration
- Testar com dados reais

**3. RAG no Chat:**
- Investigar se `ChatFolder` usa DB ou localStorage
- Adicionar visibilidade de docs também em `ChatFolder`

**4. Validação:**
- Garantir que doc deletado não quebra RAG antigo
- Testar busca com múltiplos docs visíveis

---

## Conclusão

A refatoração é **viável e recomendada**. O sistema está bem desenhado; esta mudança generaliza o conceito de "documentos por contexto" sem quebrar o existente. Opção A (remover `notebook_id`, criar tabelas de vinculação) é a rota mais limpa.

**Recomendação:** Passar para design e desenvolvimento com atenção à migration e clareza da UI.
